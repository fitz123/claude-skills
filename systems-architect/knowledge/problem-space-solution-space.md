## Определение

**Problem Space** — пространство проблем, которые решает бизнес; описывается через поддомены. **Solution Space** — пространство решений этих проблем; описывается через боундед-контексты. Это фундаментальное разделение DDD: сначала понять проблему, потом думать о решении.

## Когда применять

- При старте любого проекта: зафиксировать, ЧТО мы решаем, до обсуждения КАК
- При именовании поддоменов и контекстов — названия должны отражать принадлежность
- При расхождении взглядов команды на архитектуру — вернуться к problem space
- При переработке legacy-системы: сначала понять оригинальные бизнес-проблемы

## Как применять

1. **Определить Problem Space**: провести интервью с бизнесом, выявить ключевые проблемы/задачи → записать как поддомены (бизнес-язык!)
2. **Проверить названия**: поддомен = бизнес-проблема («продажа продукции», «управление складом»), НЕ техническое решение
3. **Перейти в Solution Space**: для каждого поддомена определить боундед-контекст
4. **Проверить маппинг**: 1 поддомен ≈ 1 контекст (отклонения требуют объяснения)
5. **Валидировать с бизнесом**: терминология должна совпадать

## Примеры

- Поддомен «продажа продукции» (problem) → Bounded Context «Checkout» (solution) ✅
- «Магазин» как контекст ❌ — это техническое название, не бизнес-проблема
- Поддомен «начисление зарплаты» → Bounded Context «Payroll» ✅
- «UserService» ❌ — это не решение бизнес-проблемы, это entity service (антипаттерн)

## Anti-patterns

- **Смешивать пространства**: называть поддомен по технологии («микросервис авторизации»)
- **Пропускать problem space**: сразу рисовать сервисы без анализа бизнес-проблем
- **Одно решение на все проблемы**: «всё это один большой сервис» без разделения по проблемам
- **Называть решение до понимания проблемы** — ведёт к entity services и распределённому монолиту
