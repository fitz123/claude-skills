## Определение

**Change Data Capture (CDC)** — Data First паттерн декомпозиции «монолита». Идея: **сначала перенести данные** в новый сервис (через стриминг событий), и только потом — переносить бизнес-логику.

Это противоположность Code First подходов (Strangler Fig, Tactical Forking). CDC лучше всего подходит, когда конфликт в характеристиках **баз данных** (нужна другая БД, другая модель данных).

Название «Change Data Capture» отражает суть: захватываем (capture) изменения данных (change) и стримим их в новый сервис.

## Когда применять

- Нужна **новая база данных** (например, реляционная → append-only для иммутабельности)
- Нужно **изменить модель данных** (schema migration)
- Нужно **интегрировать CQRS** (разделить базы чтения и записи)
- При наличии **legacy данных**, которые нужно перенести в новую БД
- Когда важнее начать с данных, чтобы избежать проблем с потерей данных при интеграции

## Как применять

### 4 шага CDC

**Шаг 1: Наполнение данными пустого сервиса**
- Настроить стриминг событий из «монолита» в новый пустой сервис
- Если данных много или есть legacy — сделать первоначальный «дамп» данных вручную
- Новый сервис пока только копит данные, никакой бизнес-логики

**Шаг 2: Перенос или реализация логики**
- Когда данные синхронизированы, реализовать бизнес-логику в новом сервисе
- Новый сервис начинает обрабатывать входящие события и менять состояние своей БД

**Шаг 3: Вызов новой логики с заполненной базой**
- Переключить трафик на новый сервис (или пустить параллельно для валидации)
- Проверить корректность работы с реальными данными
- Бонус: можно запускать старую и новую реализацию параллельно для сравнения результатов (Shadow Mode)

**Шаг 4: Удаление старой функциональности из «монолита»**
- После подтверждения корректности удаляем старый код

### Выбор паттерна декомпозиции: 4 вопроса

```
1. Нужна ли новая база данных?        ─┐
2. Нужно ли менять модель данных?     ─┤ Если ДА → Change Data Capture
3. Надо ли использовать другой язык?  ─┤ Если НЕТ новой БД/модели:
4. Нужно ли менять бизнес-процессы?  ─┘   → Strangler Fig (другой язык) 
                                           → Tactical Forking (тот же язык)
```

## Плюсы и минусы

**Плюсы:**
- **Заставляет делать самое сложное сразу** — интеграция. Не даёт забыть про неё до дедлайна
- **Rollback стратегия**: старый сервис работает параллельно, можно откатиться
- **Shadow Mode**: запускаем оба сервиса, сравниваем результаты — отличный способ валидации
- Позволяет постепенно переходить без остановки системы

**Минусы:**
- **Скрипты для перегона** legacy данных отнимают время
- **Риск потери данных** в момент переноса (если не настроить правильно)
- Требует понимания **event-driven архитектуры** (Kafka, RabbitMQ, etc.)
- Нужен глубокий анализ бизнес-процессов для определения корректных границ

⚠️ **Предупреждение Антона:** Менять структуру БД и переписывать бизнес-логику одновременно — в 99% случаев вызывает боль. Лучше делать по одному шагу. Пример: распил сервиса расчёта цен занял полгода вместо месяца, потому что сменился и алгоритм, и БД — цены расходились в пределах 5%, точное сравнение не работало.

## Примеры

### Happy Cat Box — расчёт зарплаты тестировщиков

**Ситуация:** финансовый регулятор требует иммутабельные данные о выплатах. Текущая БД не поддерживает. Нужна другая БД (append-only).

**Решение через CDC:**
1. Настроить стриминг событий о выплатах из монолита → пустой новый сервис с append-only БД
2. Перегнать исторические данные о выплатах вручную
3. Реализовать бизнес-логику расчёта зарплаты в новом сервисе
4. Включить Shadow Mode: оба сервиса считают зарплату, сравниваем результаты
5. После валидации — переключить трафик, удалить старую логику из монолита

### КИБЕР-ГЕНПОДРЯД

**Сценарий:** процентовка (КС-2, КС-3) требует иммутабельных финансовых данных (audit trail для ФЗ-44).

**CDC подход:**
1. Настроить стриминг событий приёмки работ из основного монолита
2. Создать отдельный сервис процентовки с append-only хранилищем (Event Sourcing)
3. Перегнать исторические акты КС-2/КС-3
4. Реализовать логику подписания и финального расчёта
5. Shadow mode: параллельный расчёт для проверки
6. Переключить бухгалтерию на новый сервис

## Связь с CQRS

CDC — идеальный способ внедрить CQRS:
- Из «монолита» стримим события → WRITE сервис (append-only, consistency)
- Из WRITE сервиса репликация → READ сервис (key-value или колоночная, availability)

## Anti-patterns

- **Начинать с логики, не с данных** — классическая ошибка. Потом страдаешь с дампами и расхождением данных
- **Игнорировать legacy данные** — стриминг настроили, но исторические данные не перенесли → новый сервис работает некорректно
- **Менять бизнес-логику одновременно со структурой данных** — двойная сложность
- **Не делать Shadow Mode** — переключение без параллельной валидации рискованно
- **CDC для простых случаев** — если не нужна новая БД и нет смены модели данных, используй Code First паттерны (проще)
