## Определение

Архитектурная ката — подготовленное упражнение для практики проектирования систем. Это описание требований к воображаемой системе, которое нужно спроектировать самостоятельно или в группе. Название происходит от японской концепции «ката» в боевых искусствах — отработки движений до автоматизма.

**Зачем нужны**: читать курсы и книги недостаточно — проектирование системы нужно практиковать. Каты дают безопасную среду для ошибок и экспериментов.

**Откуда брать**: оригинальный сайт от Ted Neward: [nealford.com/katas](https://nealford.com/katas/), архитектурные каты от O'Reilly. Также чужие решения можно найти на GitHub.

## Когда применять

- Для самостоятельного обучения архитектурному мышлению
- Для подготовки к архитектурным интервью
- Для командных воркшопов и обмена опытом
- Когда нет возможности практиковаться на реальных проектах

## Как применять

### Алгоритм работы с катой

1. **Открыть рандомную кату** — взять случайную, не выбирать «знакомый домен»
2. **Прочитать требования** внимательно, выписать вопросы
3. **Применить полный чек-лист проектирования**:
   - Определить стейкхолдеров
   - Собрать функциональные и нефункциональные требования
   - Выявить ограничения
   - Провести Event Storming
   - Определить поддомены и контексты
   - Расставить характеристики
   - Выбрать архитектурный стиль
   - Задокументировать решения (ADR + диаграммы)
4. **Описать спорные места** в отдельном файле — где не уверены, почему именно так
5. **Провалидировать с коллегами или друзьями** — показать, получить обратную связь
6. **Изучить чужие решения** той же каты — расширить насмотренность

### Форматы проведения

**Индивидуально:**
- Выделить 2-4 часа
- Работать как будто это реальный проект (документировать, делать диаграммы)
- Не смотреть чужие решения до завершения

**В группе:**
- Разбиться на команды по 2-3 человека
- Каждая команда проектирует независимо
- Презентация и сравнение решений в конце
- Обсуждение trade-offs разных подходов

### Ресурсы для поиска кат

- **Neal Ford's Architecture Katas**: http://nealford.com/katas/
- **O'Reilly Architecture Katas**: catalog на сайте O'Reilly
- **GitHub**: искать `architecture kata solutions` — чужие решения для насмотренности
- **Чат курса**: публиковать свои решения, получать фидбек

## Примеры

### Пример структуры описания каты (что обычно даётся):
- **Название**: "Online Auction System"
- **Контекст**: стартап хочет запустить платформу онлайн-аукционов
- **Пользователи**: 500 одновременных, рост до 10K через год
- **Требования**: 
  - Realtime ставки
  - Уведомления при перебивании ставки
  - Интеграция с платёжной системой
- **Ограничения**: 
  - Бюджет: $50K в год на инфраструктуру
  - Команда: 5 разработчиков

### Мини-ката: КИБЕР-ГЕНПОДРЯД как учебное упражнение

КИБЕР-ГЕНПОДРЯД сам по себе отличная ката для практики:

**Данные:**
- Строительная платформа для управления объектами
- Пользователи: рабочий, ИТР, прораб, РПП, РПГ
- Ключевые процессы: ГПР → приёмка работ → процентовка
- Документы: КС-2, КС-3
- Ограничения: работа в поле при плохом интернете

**Что нужно спроектировать:**
1. Bounded contexts и их границы
2. Модель данных для иерархии «Объект → Этап → Работа»
3. Архитектурный стиль с обоснованием
4. Стратегию offline-first для мобильного клиента

## Anti-patterns

- **Выбирать только знакомые домены**: ката — это выход из зоны комфорта. Незнакомый домен → более честная практика
- **Пропускать документирование**: «я знаю решение, зачем его записывать» → теряется навык коммуникации через документы
- **Смотреть чужие решения сразу**: лишает себя самостоятельного мышления
- **Работать без валидации**: решение без фидбека → не развивается критическое мышление
- **Kata ≠ production**: переносить паттерны из кат напрямую без учёта реального контекста
