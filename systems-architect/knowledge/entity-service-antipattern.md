## Определение

Два классических антипаттерна при разбиении системы на элементы:

1. **Entity Service** — выделение сервиса вокруг сущности данных (User Service, Product Service, Price Service), а не вокруг бизнес-процесса
2. **Технические шаги** — выделение отдельного сервиса для технических операций (нотификации, валидация, логирование), которые являются частью бизнес-команды, а не самостоятельным процессом

Оба приводят к высокому coupling и распределённому монолиту.

## Когда применять (как их распознать)

**Признаки Entity Service:**
- Название сервиса = название сущности: «UserService», «ToyService», «PriceService»
- Сервис не отвечает на вопрос «что делает система», а отвечает на «чем владеет»
- Любые изменения данных ломают половину системы (высокий afferent coupling)

**Признаки Технических шагов:**
- В ES-модели цепочка из 3+ событий, связанных стрелками — скорее всего это один бизнес-процесс
- Нотификация вызывается после каждого события → она технический шаг, не контекст
- Название сервиса/события описывает «как» происходит, а не «что произошло»: «ValidateData», «ProduceEvent», «StoreRecord»

## Как избежать

**Вместо Entity Service:**
- Декомпозировать по бизнес-процессам (bounded contexts), не по сущностям
- Спросить: «Какой бизнес-процесс этот контекст обслуживает?»
- Одна бизнес-проблема → один контекст. Не одна сущность → один сервис

**Вместо Технических шагов:**
- В ES не описывать технические детали реализации команды
- Если несколько стрелочно-связанных событий → проверить, можно ли объединить в одну бизнес-команду
- Нотификация = последний шаг команды, не отдельный сервис (если только это не ключевой бизнес-процесс)

**Тест на антипаттерн:**
- Имя сервиса в терминах бизнеса? ✅ («Оформление заказа») или в технических? ❌ («OrderService»)
- Событие отвечает на «что произошло»? ✅ («Заказ оплачен») или «как произошло»? ❌ («PaymentProcessed via Stripe»)
- DRY-нарушение: два сервиса вызывают один и тот же код? → лучше продублировать код, чем создавать зависимость

## Примеры

**Entity Service — ошибка:**
- Выделили «Price Service» для хранения цен → любое изменение отображения цены ломает все запросы клиентов по всей системе → 2 года разработки, ничего не ускорило
- «Notification Service» стал точкой отказа для мобильного приложения → без уведомлений не работает ничего

**Entity Service — правильно:**
- Вместо «UserService» — контексты «Регистрация», «Биллинг», «Профиль» — каждый отвечает за свой процесс

**Технические шаги — ошибка:**
- Команда «Оформить заказ» разбита на: «Validate» → «StoreOrder» → «SendNotification» → «UpdateInventory» — 4 отдельных сервиса
- Любое изменение требует координации 4 сервисов одновременно

**Технические шаги — правильно:**
- Одна команда «Оформить заказ» содержит все шаги внутри. В ES — одно событие «Заказ оформлен»

**КИБЕР-ГЕНПОДРЯД (строительство):**
- ❌ «NotificationService» для рассылки прорабам: «смена назначена», «акт подписан», «оплата произведена» → стал зависимостью для всего
- ✅ Уведомление = шаг команды. «Назначить смену» → внутри команды: сохранить, уведомить SMS
- ❌ «WorkerService» — entity service для рабочих: любое изменение модели рабочего ломает ГПР, Приёмку и Финансы одновременно
- ✅ Разные контексты: «Управление персоналом» (создаёт рабочих), «ГПР» (читает, назначает), «Приёмка» (читает подпись)

## Anti-patterns (anti-anti-patterns)

- **«Нотификация всегда — технический шаг»**: если отправка SMS является ключевым бизнес-процессом (например, OTP-верификация), она может быть отдельным контекстом
- **«DRY важнее изоляции»**: дублирование кода в разных сервисах лучше, чем общая зависимость. Если два сервиса вызывают одну функцию — это признак слишком тонкой декомпозиции или entity service
- **«Чем больше сервисов, тем лучше»**: каждый сервис — операционная нагрузка. Оправдан только когда бизнес-процессы действительно независимы
