## Определение

Выбор **типа** базы данных — архитектурное решение, основанное на характеристиках системы и природе данных. Задача архитектора — выбрать тип (реляционная, документная, графовая...), а конкретную технологию (PostgreSQL vs MySQL) выбирают разработчики с учётом своей экспертизы, доступности в облаке и других тактических факторов. Выбор "MongoDB потому что модно" — архитектурная ошибка.

## Когда применять

- При проектировании нового сервиса или системы
- При решении о разделении на микросервисы (каждый сервис выбирает свой тип БД)
- Когда текущая БД не справляется с характеристиками (масштабирование, консистентность)

## Как применять

**Таблица выбора типа БД по характеристикам** (из Fundamentals of Software Architecture, гл. 6):

| Тип БД | Сильные стороны | Слабые стороны | Когда выбирать |
|---|---|---|---|
| **Реляционная** (PostgreSQL, MySQL) | Consistency, ACID-транзакции, сложные JOIN | Масштабирование записи, слабоструктурированные данные | Финансы, инвентарь, строгая consistency |
| **Документная** (MongoDB, CouchDB) | Гибкая схема, горизонтальное масштабирование | Нет ACID-транзакций между документами, дубликация | Каталоги, CMS, нет строгой схемы |
| **Ключ-значение** (Redis, DynamoDB) | Очень быстрое чтение, масштабируемость | Нет сложных запросов, нет связей | Кэш, сессии, счётчики, простые данные |
| **Графовая** (Neo4j, ArangoDB) | Сложные связи, траверс графов | Нет строгой consistency, узкоспециализирована | Рекомендации, социальные сети, слабоструктурированные связанные данные |
| **Колоночная** (Cassandra, HBase) | Высокий throughput записи, масштабируемость | Нет ACID, сложные запросы ограничены | Логи, временные ряды, аналитика |
| **Time-series** (InfluxDB, TimescaleDB) | Оптимизирована под временные ряды | Узкая специализация | Метрики, IoT, мониторинг |
| **NewSQL** (CockroachDB, Spanner) | ACID + горизонтальное масштабирование | Сложность, стоимость | Нужна consistency И масштабируемость |

**Алгоритм выбора:**

1. **Определить природу данных**: структурированные? иерархические? граф связей?
2. **Определить ключевые характеристики**: consistency vs. availability (CAP), write vs. read heavy
3. **Посмотреть на транзакционные требования**: нужны ACID-транзакции между сущностями?
4. **Оценить нагрузку**: объём, скорость роста, паттерны запросов (OLTP vs. OLAP)
5. **Учесть внешние ограничения**: compliance (иммутабельность), облачный провайдер, экспертиза команды
6. **Зафиксировать в ADR** с обоснованием выбора

**Важный нюанс про CAP-теорему**: для одного и того же семейства БД существуют реализации под разные CAP-компромиссы. Например, среди column-oriented есть CA, CP и AP базы. Всегда проверяйте конкретную технологию, а не только семейство.

## Примеры

**Happy Cat Box:**
- Биллинг (финансовый комплаенс + иммутабельность) → **реляционная** (отдельная изолированная)
- Определение параметров игрушек (слабоструктурированные связанные данные) → **графовая**
- Магазин и мэтчинг (consistency при заказах) → **реляционная** (изолированная)
- Остальные сервисы → **реляционная** (по умолчанию: структурированные данные + transactions)

**КИБЕР-ГЕНПОДРЯД:**
- Выполнение работ (история, иммутабельность, КС-2) → **реляционная** + event sourcing подход
- ГПР (иерархическая структура работ + связи) → **реляционная** с ltree/adjacency list ИЛИ **документная**
- Аналитика и дашборды прораба → **колоночная** или read-модель (CQRS)
- Уведомления/очереди задач → **ключ-значение** (Redis) как буфер

## Anti-patterns

- **"MongoDB потому что модно"**: выбор технологии без анализа характеристик
- **Одна БД на всю систему**: если сервисы имеют разные характеристики — нужны разные типы
- **Реляционная везде "на всякий случай"**: если данные иерархические или слабоструктурированные — переплата за JOIN
- **Игнорировать экспертизу команды**: самая подходящая БД бесполезна, если никто не умеет с ней работать
- **Забыть про compliance**: финансовые данные могут требовать конкретного типа хранения по закону
- **Выбирать по статьям 2010 года**: рынок БД быстро меняется, проверяйте актуальность
