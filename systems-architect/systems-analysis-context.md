# Системный анализ — Knowledge Base для проектирования

> Сжатый справочник на основе курса «Анализ систем» (Школа Сильных Программистов). Содержит практически применимые концепции для проектирования архитектуры ИТ-систем. Используется как контекст для AI-агента.

---

## 1. Ключевые концепции

### Система = элементы + связи
Система — набор элементов и их взаимосвязей, где функциональность целого превышает сумму частей. У системы есть **форма** (структура данных, код, интерфейсы) и **функция** (поведение, процессы). Связи бывают формальные (общие данные) и функциональные (один процесс запускает другой). Чтобы найти все связи, нужно исследовать оба вида: функциональные — через Event Storming, формальные — через модель данных.

### Problem Space vs Solution Space
**Problem Space** — набор проблем, которые решает бизнес. **Solution Space** — набор решений этих проблем. Поддомены описывают проблемы, боундед-контексты — решения. Названия должны это отражать: «продажа продукции» (поддомен/проблема), а не «магазин» (техническая реализация).

### Стратегия vs Тактика
**Стратегия** — выбор способа достижения цели (какие элементы, как связаны, какой стиль). **Тактика** — конкретные решения в рамках стратегии (какой фреймворк, какая БД). Архитектор отвечает за стратегию и задаёт ограничения; дизайн (тактику) делегирует команде. Описывать «что и почему» вместо «как» — ключевой принцип передачи решений разработчикам.

### Coupling и Cohesion
**Coupling** (внешняя связанность) — связи между элементами системы. **Cohesion** (внутренняя связанность) — связи внутри одного элемента. Стремимся к **низкому coupling** и **высокому cohesion**, чтобы изменения в одном элементе не ломали другие. Но coupling = 0 невозможен — иначе это не система, а набор несвязанных частей.

**10 видов coupling** (для развития насмотренности):
- **Temporal** — строгий порядок выполнения (регистрация → оплата)
- **Functional** — одинаковая бизнес-логика в нескольких местах
- **Implementation** — общая техническая реализация
- **Static** — связи по данным (FK в БД, общие модели)
- **Dynamic** — вызовы в рантайме (HTTP, RPC, события)
- **Afferent/Efferent** — входящие/исходящие вызовы (для расчёта instability)
- **Through resources** — общие ресурсы (CPU, память, файлы, БД)
- **Semantic** — общий протокол/формат данных
- **Deployment** — зависимости при деплое

**Когда применять:** при анализе ES-модели и модели данных ищите все виды связанности. Если два контекста имеют temporal coupling + through resources coupling, вероятно это один контекст.

### Local vs Global Complexity
Сложность бывает **локальной** (внутри элемента) и **глобальной** (между элементами). Микросервисы уменьшают локальную сложность, но увеличивают глобальную. Монолит — наоборот. Выбор между ними — это trade-off, а не «серебряная пуля».

## 2. Методология проектирования систем

### Три этапа архитектурного решения
Любое архитектурное решение проходит три этапа: **сбор требований → анализ → принятие решений**. В конце каждого этапа — обязательная валидация со стейкхолдерами.

### Рекомендуемая последовательность работы

**Этап 1: Сбор требований**
1. Определить стейкхолдеров и приоритизировать их (матрица «Влияние — Интерес»)
2. Собрать консёрны стейкхолдеров (требования, ограничения, опасения)
3. Собрать функциональные требования (user stories, поведение системы)
4. Найти внешние ограничения (законы, бюджет, уровень команды, регуляции)
5. **Валидация** со стейкхолдерами

**Этап 2: Анализ**
1. Определить бизнес-домен и поддомены (Strategic DDD)
2. Классифицировать поддомены: Core / Generic / Supporting (Core Domain Chart)
3. Определить боундед-контексты через поддомены
4. Сделать Event Storming — найти поведение системы
5. Сделать модель данных — найти структуру системы
6. Сравнить контексты из ES с контекстами из DDD, найти расхождения
7. Определить архитектурные характеристики (явные и неявные)
8. **Валидация** со стейкхолдерами

**Этап 3: Принятие решений**
1. Выбрать архитектурный стиль на основе характеристик
2. Выбрать виды коммуникаций (синхронные/асинхронные)
3. Выбрать виды БД для каждого сервиса
4. Определить паттерны для достижения нужных характеристик
5. Описать решения через ADR
6. Определить fitness functions для верификации
7. **Валидация** со стейкхолдерами

### Работа с требованиями — практические приёмы

- **Нет требований?** Напишите сами и дайте отревьюить — это провоцирует обратную связь
- **Спрашивайте «Зачем?» и «Почему?»**, а не «Что?» и «Как?» — копайте до корневой проблемы
- **Уточняйте абстракции**: «быстро» → «за 200мс», «много пользователей» → «10K одновременных»
- **Ведите глоссарий** бизнес-терминов
- **Изучайте state transitions** — переходы состояний сущностей (заказ: создан → оплачен → собран → доставлен)
- **Думайте об ошибках**: что если платёж не прошёл? что если данные не валидны?
- **Источники требований при их отсутствии**: документация, исходный код, аналогичные проекты, работа «в полях», мозговой штурм, опросники

## 3. Стратегический DDD

### Business Domain
Главная проблема/цель бизнеса. Определяется через бизнес-канвас или беседу с руководством.

### Поддомены (Sub-domains)
Составные проблемы бизнес-домена. Стартовая точка поиска — департаменты компании, но один департамент может решать несколько проблем. Разбивайте по бизнес-функциям.

**Три типа поддоменов:**

| | Core | Generic | Supporting |
|---|---|---|---|
| Конкурентное преимущество | Да | Нет | Нет |
| Сложность модели | Высокая | Любая | Низкая |
| Изменчивость | Частая | Редкая | Редкая |
| Реализация | Инхаус, лучшие люди | Купить/аутсорс | Инхаус/аутсорс/джуны |

**Быстрый тест типа поддомена:**
1. Можно ли купить готовое решение и компания останется собой? → Generic
2. Справятся ли джуны без контроля качества? → Supporting
3. Нет к обоим? → Core

**Core Domain Chart** — визуализация на осях «Сложность модели» × «Влияние на бизнес». Помогает увидеть приоритеты и договориться с бизнесом.

### Ubiquitous Language (Общий язык)
Единый язык для бизнеса и разработки в рамках боундед-контекста. Включает не только термины, но и описание процессов. Описывается через глоссарий, ES-модели, тесты на Gherkin, sequence diagrams.

### Bounded Context (Боундед-контекст)
Изолированная область, в которой общий язык однозначен. Каждый контекст решает конкретную проблему.

**Правила:**
- 1 поддомен ≈ 1 боундед-контекст (стремиться, но не догма)
- 1 боундед-контекст = 1 команда разработки (обязательно; одна команда может владеть несколькими)
- Если 1 контекст → много поддоменов — он слишком большой, дели
- Если 1 поддомен → много контекстов — вероятно перерезали, проверяй

**Типичные ошибки:**
- Называть контекстом техническую сущность («биллинг» — часто это 2 контекста: оплата клиентом и выплата подрядчику)
- Разбивать единый бизнес-процесс на контексты без причин по характеристикам (чекаут = корзина + оплата — плохо)
- Искать «идеальные» границы в одиночку — нужен фидбэк команды

## 4. Архитектурные фреймворки и стили

### Описание архитектуры
ADR описывает решения, но не архитектуру целиком. Для полного описания используйте фреймворки:
- **4+1** — 5 представлений системы (logical, process, development, physical + scenarios)
- **C4** — 4 уровня детализации (Context, Container, Component, Code)
- **arc42** — популярный шаблон документа архитектуры (12 секций)
- **ISO 42010** — стандарт Architecture Description

Для веб-проектов обычно хватает: общая структура (C4 Level 1-2), поведение (sequence diagrams), характеристики элементов, коммуникации.

### 7 архитектурных стилей

**Монолитные:**

| Стиль | Суть | Когда |
|---|---|---|
| **Layered** (N-layer) | Слои: presentation → business → persistence | Простое приложение, мало денег, старт |
| **Modular Monolith** | Модули по боундед-контекстам, одна БД | Нужна модульность без сложности распределённых систем |
| **Microkernel** (Plugin) | Ядро + плагины | Часто меняющаяся логика (алгоритмы, правила), IDE, ETL-пайплайны |
| **Pipeline** | Цепочка фильтров-трансформаций | Последовательная обработка данных (map-reduce, ETL) |

**Распределённые:**

| Стиль | Суть | Когда |
|---|---|---|
| **Service-Based** | Крупные сервисы + общая БД | Промежуточный между монолитом и микросервисами, самый прагматичный распределённый стиль |
| **Microservices** | Мелкие сервисы, каждый со своей БД | Нужны противоречивые характеристики в одном сервисе ИЛИ инфраструктурная изоляция кода |
| **Event-Driven (EDA)** | Коммуникация через события | Высокая расцепленность, масштабируемость, eventual consistency допустима |

### Сравнение стилей по характеристикам (★ = лучше)

| Характеристика | Layered | Service-Based | Microservices |
|---|---|---|---|
| Simplicity/Cost | ★★★★★ | ★★★ | ★ |
| Deployability | ★ | ★★★★ | ★★★★★ |
| Scalability | ★ | ★★★ | ★★★★★ |
| Fault Tolerance | ★ | ★★★★ | ★★★★★ |
| Modularity | ★ | ★★★★ | ★★★★★ |
| Performance | ★★★ | ★★★ | ★★ |
| Testability | ★★ | ★★★★ | ★★★★ |

**Ключевой вывод:** микросервисы оправданы в двух случаях: (1) нужны противоречащие характеристики внутри одного сервиса; (2) нужна инфраструктурная изоляция кода. Во всех остальных случаях service-based дешевле и проще.

## 5. Работа с характеристиками (Quality Attributes)

### 12 ключевых характеристик

| # | Характеристика | Описание |
|---|---|---|
| 1 | **Availability** | Устойчивость к отказам, % uptime |
| 2 | **Scalability** | Способность расти под нагрузкой |
| 3 | **Modifiability** | Цена и риск изменений |
| 4 | **Maintainability** | Сложность восстановления и обслуживания |
| 5 | **Security** | Защита от несанкционированного доступа |
| 6 | **Performance** | Время отклика на действия |
| 7 | **Agility** | Гибкость — умение подстраиваться под изменения |
| 8 | **Testability** | Полнота и простота тестирования |
| 9 | **Deployability** | Скорость и простота выкатки |
| 10 | **Usability** | Удобство для пользователей |
| 11 | **Consistency** | Все видят одинаковые данные в одно время |
| 12 | **Simplicity** | Понятность системы |

### Как находить характеристики

**Явные** — прямо из требований (числа пользователей, RPS, uptime SLA).

**Неявные** — из 5 источников:
1. **Фразы-маркеры** от бизнеса: «time to market» → agility, testability, deployability; «удовлетворённость» → performance, availability; «бюджет» → simplicity
2. **Требования** — зависимости между элементами, интеграции, безопасность
3. **Цели компании** — core домены нуждаются в availability, modifiability, maintainability
4. **Домен** — финтех → security+consistency; медицина → compliance+privacy
5. **Этап жизни компании**: стартап → simplicity, modifiability; рост → scalability, agility; зрелость → maintainability, reliability

### Quality Attribute Scenarios
Для «абстрактных» характеристик (usability, maintainability) — описывайте сценарии:
- ❌ «Высокий usability»
- ✅ «Новый администратор добавляет пользователя менее чем за 10 минут»
- ✅ «Фича выкатывается максимум за 2 недельных спринта»

Каждый сценарий должен быть **тестируемым**.

## 6. Интеграции и взаимодействие компонентов

### Выбор коммуникаций
Выбирайте вид коммуникации на основе характеристик, а не привычки:
- **Синхронные (HTTP/gRPC)** — простота, consistency, но coupling выше
- **Асинхронные (события, очереди)** — decoupling, scalability, но eventual consistency и сложность

### Event Storming — нотация
6 базовых стикеров:

| Цвет | Роль | Правило |
|---|---|---|
| Жёлтый | **Актор** | Кто запускает команду (человек, система, крон) |
| Голубой | **Команда** | Что делает система (настоящее время, бизнес-термины) |
| Оранжевый | **Событие** | Что произошло (прошедшее время, необратимо) |
| Фиолетовый | **Policy** | Условие автоматического запуска команды |
| Розовый | **Внешняя система** | Другая система (эквайринг, крон, API) |
| Зелёный | **Read Model** | Данные для принятия решения актором |

**Ключевые правила ES:**
- События — бизнесовые, не технические: «Заказ оплачен» ✅, «Record created in DB» ❌
- Команды — «что», а не «как»: «Checkout» ✅, «Validate params» ❌
- Policy описывает бизнес-контекст, не техническую валидацию
- Цепочка из 3+ связанных стрелками событий — вероятно, технические шаги одной команды
- Внешняя система всегда возвращает событие, вызывается из команды
- Крон/таймер — это внешняя система (scheduler)

### Модель данных
Концептуальная (не схема БД!): какие данные нужны бизнесу и как связаны.

**3 шага:**
1. Перенести контексты из ES как пустые блоки
2. Заполнить данными каждый блок (дублировать общие сущности в разных контекстах)
3. Определить владельцев данных (кто создаёт/мутирует → owner, кто только читает → consumer), показать стрелками owner → consumer

**Красные флаги:** несколько владельцев одних данных → проблемы с консистентностью; много связей между двумя контекстами → вероятно один контекст.

### Выбор БД по характеристикам
Не выбирайте конкретную технологию — выбирайте **тип БД** по характеристикам:
- **Реляционная** — consistency, сложные связи данных, транзакции
- **Документная** — flexibility, rapid development, неструктурированные данные
- **Key-Value** — performance, кэширование, сессии
- **Графовая** — сложные связи между сущностями, рекомендации
- **Колоночная** — аналитика, агрегации больших объёмов
- **Time-series** — метрики, IoT, логи

### Паттерны коммуникаций
- **Strangler Fig** — постепенная замена старой системы новой (вместо Big Bang)
- **CQRS** — разделение чтения и записи (когда read/write характеристики противоречат)
- **Event Sourcing** — хранение истории изменений как последовательности событий
- **Saga** — распределённая транзакция через цепочку локальных транзакций

## 7. Стейкхолдеры и ограничения

### Матрица стейкхолдеров
Приоритизация по осям **Влияние × Интерес**:
1. ↑Влияние + ↑Интерес → **Управляй тесно** (ключевые стейкхолдеры, валидируй решения)
2. ↑Влияние + ↓Интерес → **Держи довольными** (учитывай, но не вовлекай глубоко)
3. ↓Влияние + ↑Интерес → **Информируй** (прислушивайся, но не обязательно выполняй)
4. ↓Влияние + ↓Интерес → **Мониторь** (дополнительный контекст)

**Типичные группы:** топ-менеджмент, менеджеры продукта, финансы, юристы, разработчики, QA, DevOps/админы, клиенты, регуляторы.

### Внешние ограничения
То, что нельзя изменить: законодательство (хранение ПД, финансовые регуляции), бюджет, уровень команды, существующая инфраструктура, SLA внешних систем, корпоративные стандарты.

## 8. Документирование решений (ADR)

### Architecture Decision Record
Текстовый документ на 1-2 страницы, фиксирующий **контекст** принятия решения.

**Обязательные поля:**
- **Title** — короткое описание + номер
- **Status** — Proposed → Accepted / Rejected / Superseded (указать номер нового)
- **Context** — условия, причины, альтернативы
- **Decision** — что решили и **почему** (не «как»)
- **Consequences** — влияние на систему, trade-offs

**Дополнительные поля:** Compliance (как проверять), Notes (метаинформация), Alternatives (анализ альтернатив).

**Правила:**
- Один ADR = одно решение
- Директивный язык: «мы будем использовать» вместо «нам кажется, что»
- Храните в git или wiki, доступ для всех
- Определите критерии, когда нужен апрув: цена решения, влияние на команды, безопасность

## 9. Верификация (Fitness Functions)

Фитнес-функция — автоматическая или ручная проверка соответствия системы заданным характеристикам.

**Примеры:**
- Покрытие тестами ≥ 90% → проверяет testability
- Время отклика API < 200ms → проверяет performance
- Релиз не реже раза в 2 недели → проверяет deployability
- Нагрузочные тесты с 10K RPS → проверяет scalability
- Статический анализ coupling/cohesion → проверяет maintainability
- Schema registry для событий → проверяет consistency

**Инструменты:** автотесты, нагрузочные тесты, статические анализаторы (ArchUnit в Java), метрики из CI/CD, метрики из трекера задач.

## 10. Рефакторинг распределённых систем (Урок 4)

### 5 подходов к модернизации
1. **Добавить новый функционал как отдельный сервис** — когда характеристики нового не совпадают с существующим
2. **Объединить технические шаги** — когда несколько «сервисов» — на самом деле шаги одного бизнес-процесса (нотификации, валидация)
3. **Переписать сервис** — когда характеристики изменились и текущая реализация не подходит
4. **Вынести из монолита** — Strangler Fig Pattern: постепенная миграция, не Big Bang
5. **Избавиться от Entity Service** — сервис вокруг сущности («UserService», «OrderService»), а не бизнес-процесса → распределённый монолит

### Метрика Instability
`I = Ce / (Ca + Ce)`, где Ca — afferent coupling (входящие), Ce — efferent coupling (исходящие).
- I = 0 — максимально стабильный (все от него зависят)
- I = 1 — максимально нестабильный (зависит от всех)

Считайте instability при добавлении/удалении сервисов, чтобы оценить каскадный эффект.

### Антипаттерн: Entity Service
Сервис, построенный вокруг сущности (UserService, ProductService), а не бизнес-процесса. Проблема: любое изменение бизнес-логики затрагивает множество «entity-сервисов» → распределённый монолит. Решение: строить сервисы вокруг боундед-контекстов (бизнес-процессов).

## 11. Чек-листы и Best Practices

### Чек-лист проектирования системы
- [ ] Определены стейкхолдеры и их консёрны
- [ ] Собраны и уточнены требования (без абстрактных формулировок)
- [ ] Найдены внешние ограничения (законы, бюджет, SLA, уровень команды)
- [ ] Определён бизнес-домен и поддомены
- [ ] Классифицированы поддомены (Core/Generic/Supporting)
- [ ] Определены боундед-контексты, проверено 1:1 с поддоменами
- [ ] Сделан Event Storming (события бизнесовые, не технические)
- [ ] Сделана модель данных (концептуальная, не схема БД)
- [ ] Найдены все виды coupling между контекстами
- [ ] Определены архитектурные характеристики (явные + неявные)
- [ ] Выбран архитектурный стиль на основе характеристик
- [ ] Выбраны виды коммуникаций (синхронные/асинхронные) с обоснованием
- [ ] Выбраны типы БД по характеристикам
- [ ] Написаны ADR для ключевых решений
- [ ] Определены fitness functions для верификации

### Чек-лист Event Storming
- [ ] События описывают бизнес, не техническую реализацию
- [ ] Команды используют бизнес-термины
- [ ] Нет цепочек из 3+ технических шагов, замаскированных под бизнес-команды
- [ ] Policy описывает бизнес-контекст, а не «данные валидны»
- [ ] Read model указан только для неочевидных данных
- [ ] Сохранена аннотация ES (порядок стикеров, связи)
- [ ] Есть легенда для всех стикеров

### Чек-лист модели данных
- [ ] Модель концептуальная (бизнес-данные), не логическая (схема БД)
- [ ] Данные продублированы в разных контекстах (а не стрелки-спагетти)
- [ ] Определены владельцы данных (кто создаёт/мутирует)
- [ ] Нет нескольких владельцев одних данных
- [ ] Линии не пересекаются (или пересечения явные)
- [ ] Есть легенда (типы стрелок, цвета)

### Практические советы
1. **Работайте публично** — все артефакты в одном месте, доступны команде
2. **Не забивайте на описание** — контекст теряется за недели; пишите «почему», а не «что»
3. **Храните версии** моделей и описаний (git, вкладки в диаграммных инструментах)
4. **Переключайтесь** если зависли — от ES к модели данных и обратно
5. **Не меняйте работающее** без разговора с руководством — исправление может быть дороже пользы
6. **На старте проекта** — максимальная изоляция каждой гипотезы, не анализ всего сразу
7. **Не предлагайте решение в первые минуты** — сначала разберитесь в проблеме
8. **Оставляйте артефакты** — чем подробнее, тем проще разобраться потом
9. **Один ADR = одно решение** (стиль и БД — разные ADR)
10. **Нотификации НЕ могут быть отдельным сервисом** — это технический шаг бизнес-команды
