## Определение

**Instability (I)** — характеристика, выражающая вероятность того, что при изменении связанных элементов придётся менять данный элемент.

**Формула:**
```
I = Ce / (Ca + Ce)
```

Где:
- **Ca (Afferent Coupling)** — входящие связи (сколько других элементов зависят от данного)
- **Ce (Efferent Coupling)** — исходящие связи (от скольких других элементов зависит данный)

**Значения:**
- `I = 0` — максимально стабильный (все только зависят от него, он ни от кого)
- `I = 1` — максимально нестабильный (сам зависит от других, от него никто)
- `0 < I < 1` — частично стабильный

> ⚠️ Instability ≠ количество отказов системы. Это вероятность необходимости изменения элемента из-за изменений в окружающей системе.

## Когда применять

- При анализе технических сервисов (нотификации, entity-сервисы)
- При проверке, не сделали ли важные core-сервисы нестабильными
- При выборе между вариантами реализации: считай instability для каждого варианта
- После добавления нового сервиса в систему — пересчитай instability связанных сервисов
- При подозрении на слишком высокую связанность

## Как применять

### Что считать связями

**Для сервисов:**
- **Ca (входящие)** = эндпоинты команд (синхронные) + консьюмеры (асинхронные)
- **Ce (исходящие)** = вызовы HTTP/RPC-клиентов + продьюсинг событий

**Пример расчёта:**
Сервис имеет 2 эндпоинта и 1 исходящий вызов:
```
Ca = 2, Ce = 1
I = 1 / (2 + 1) = 0.33 → стабильный
```

### Нюанс: синхронные вызовы

При синхронных коммуникациях нужно разделять **две системы**:

1. **Система с учётом схемы данных** — оба сервиса зависят от схемы (контракта). У обоих instability ≈ 0.5.
2. **Система только с бизнес-логикой** — смотришь на логику, не на контракт. Например, сервис заказа не меняется при смене алгоритма расчёта маршрута.

### Алгоритм анализа

1. Нарисуй все связи между элементами (стрелки)
2. Для каждого элемента посчитай Ca и Ce
3. Вычисли I = Ce/(Ca+Ce)
4. Выдели **важные** элементы системы (core-поддомены)
5. Проверь: важные элементы должны стремиться к I=0

## Примеры

### Сервис нотификаций (Happy Cat Box)

**Вариант A: отдельный технический сервис нотификаций**
- Сервис нотификаций: Ca=4, Ce=0 → I = 0/(0+4) = **0** (стабильный)
- Каждый бизнес-сервис: Ca=0, Ce=1 → I = 1/(1+0) = **1** (нестабильный!)

Проблема: core-сервисы стали нестабильными ради технического — неправильно.

**Вариант B: логика нотификаций в каждом bounded context**
- Каждый сервис: Ca=0, Ce=0 → I = 0/(0+0) = **0** (стабильный)

Вывод: Вариант B правильный — важные сервисы стабильны.

### Entity-сервис игрушек (Happy Cat Box)

- Entity-сервис: Ca=1 (склад мутирует), Ce=5 (отдаёт данным 5 сервисам) → I = 5/(1+5) = **0.83** (нестабильный!)
- Это опасно: нестабильный элемент в середине цепочки — при его поломке рушатся магазин, склад, расчёт параметров

### КИБЕР-ГЕНПОДРЯД пример

Модуль приёмки работ:
- Получает данные от: рабочего (фиксация), ИТР (верификация) → Ca=2
- Вызывает: сервис КС-2 (генерация документа), биллинг (процентовка) → Ce=2
- I = 2/(2+2) = **0.5** — нужно смотреть, можно ли уменьшить исходящие связи

## Anti-patterns

- **Гнаться за I=0 для всей системы** — невозможно. Всегда будут нестабильные элементы. Цель: важные core-элементы → I≈0.
- **Считать instability один раз** — пересчитывай при каждом изменении архитектуры.
- **Игнорировать скрытые связи** — синхронные вызовы создают неочевидные зависимости по схеме данных.
- **Путать с fault tolerance** — высокая instability не означает, что сервис упадёт. Означает, что его придётся менять при изменении окружения.
- **Не считать instability для технических сервисов** — именно они чаще всего создают нестабильность важных сервисов.
