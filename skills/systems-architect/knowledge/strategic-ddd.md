## Определение

Стратегический DDD — набор инструментов Domain-Driven Design для определения крупных блоков системы на уровне бизнеса: бизнес-домен, поддомены, боундед-контексты, общий язык. В отличие от тактического DDD (агрегаты, репозитории), стратегический отвечает на вопрос «какие части системы нам нужны и зачем».

## Когда применять

- При старте нового продукта
- При масштабировании команды (нужно определить владение)
- При разбиении монолита на сервисы
- Всегда — перед тем как рисовать сервисную архитектуру

## Как применять

1. **Определить бизнес-домен** — главная проблема/цель бизнеса (через бизнес-канвас или разговор с CEO)
2. **Выявить поддомены** — составные части домена. Стартовая точка: департаменты компании. Но делить нужно по бизнес-функциям, не по оргструктуре
3. **Классифицировать поддомены** по матрице Core/Generic/Supporting → Core Domain Chart
4. **Определить боундед-контексты** — изолированные области с собственным языком
5. **Проверить маппинг** поддомен:контекст
6. **Задать Ubiquitous Language** — глоссарий для каждого контекста
7. **Описать Context Map** — как контексты взаимодействуют

## Примеры

- Маркетплейс: домен = «торговая платформа»; поддомены = продажи, склад, логистика, финансы; контексты = Checkout, Inventory, Shipping, Billing
- Fintech: домен = «платёжный сервис»; Core = процессинг платежей; Generic = авторизация пользователей; Supporting = отчётность

## Anti-patterns

- **DDD без бизнес-знаний**: моделировать домен без участия domain expert'ов
- **Тактический DDD без стратегического**: придумывать агрегаты до понимания границ системы
- **Копировать оргструктуру**: 1 департамент = 1 поддомен — не всегда верно, один отдел может решать несколько проблем
- **Игнорировать Strategic DDD в пользу «технической архитектуры»**
