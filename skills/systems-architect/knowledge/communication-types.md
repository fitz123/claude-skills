## Определение

Выбор вида коммуникации между сервисами — стратегическое решение, влияющее на coupling, consistency и отказоустойчивость. **Синхронные** (HTTP/gRPC): сервис ждёт ответа. **Асинхронные** (события/очереди): сервис отправляет сообщение и продолжает работу. Выбор делается на основе характеристик, а не привычки.

**Три стиля коммуникаций** (из Software Architecture: The Hard Parts):

| Стиль | Описание | Реализация |
|---|---|---|
| **Request-response** | Запрос + ожидание ответа | HTTP/REST, gRPC (синхронно); async request-reply (асинхронно) |
| **Event-driven** | Продьюсер публикует событие о том, что случилось в мире; консьюмеры реагируют | Kafka, RabbitMQ, SNS/SQS (асинхронно) |
| **Common data** | Элементы не вызывают друг друга, а используют общее хранилище | Общая БД, файловая система, shared cache |

Синхронный = только request-response. Асинхронным может быть любой из трёх стилей.

**Важный нюанс: передача характеристик при синхронных вызовах**

При синхронном вызове сервис-клиент "передаёт" своё требование к latency на сервис-сервер. Если LMS должен ответить за 1 секунду и для этого синхронно вызывает сервис студентов — у сервиса студентов тоже остаётся ≤1 секунды. Это неявный coupling по характеристикам, который легко упустить.

## Когда применять

**Синхронные (HTTP/gRPC):**
- Нужен немедленный ответ (пользователь ждёт)
- Strong consistency обязательна
- Простота важнее производительности
- Внешние API (клиент-сервер)

**Асинхронные (события/очереди):**
- Eventual consistency допустима
- Нужна высокая расцепленность
- Высокая нагрузка (брокер как буфер)
- Много получателей одного события
- Долгосрочные процессы (фоновая обработка)

## Как применять

1. **Определить требование к консистентности**: нужна ли немедленная? → синхронные
2. **Оценить coupling**: хотите изолировать сервисы? → асинхронные
3. **Определить нагрузку**: есть пики? → асинхронные (брокер сглаживает)
4. **Выбрать протокол**: HTTP (простота), gRPC (производительность, типизация), Kafka/RabbitMQ (события)
5. **Зафиксировать в ADR** с обоснованием

## Примеры

- Проверка баланса при оплате → **синхронный** (нужен ответ немедленно)
- Отправка email после оплаты → **асинхронный** (не блокирует пользователя)
- Обновление аналитики → **асинхронный** (eventual consistency достаточна)
- gRPC между internal сервисами с высокой нагрузкой: меньше overhead чем HTTP/JSON

## Anti-patterns

- **Синхронные цепочки**: A→B→C→D синхронно → temporal coupling + каскадные отказы
- **Async там где нужна consistency**: оплата через события без гарантий доставки → потерянные транзакции
- **Смешивать без стратегии**: часть через HTTP, часть через Kafka без логики
- **Игнорировать idempotency** в async: дубли сообщений → дубли операций
