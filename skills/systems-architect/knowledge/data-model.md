## Определение

Концептуальная модель данных — визуальное описание того, какие данные использует система и как они связаны между собой. Не схема БД (это логическая модель), а бизнес-смысл данных без привязки к конкретному хранилищу. Отвечает на вопрос: «Что нужно бизнесу из данных для работы?»

Вместе с Event Storming формирует полную модель системы: ES описывает **функцию** (поведение), модель данных — **форму** (структуру). Объединение двух моделей показывает все связи между элементами — и по поведению, и по данным.

## Когда применять

- После Event Storming: проверить, нет ли лишних связей между выбранными контекстами
- При проектировании новой системы: до кода, после понимания требований
- При разбиении монолита: найти, кто владеет данными, чтобы правильно разрезать
- При выявлении скрытого coupling: несколько контекстов используют одни данные

## Как применять — 3 шага

### Шаг 1: Перенести элементы из ES-модели
- Взять все контексты (группы), определённые в Event Storming
- Нарисовать как пустые квадраты (без связей пока)
- **Не тащить связи из ES** — получится нечитаемое спагетти

### Шаг 2: Определить данные для каждого элемента
- Для каждого контекста записать нужные объекты данных (сущности)
- Если один объект (например, «Игрушка») нужен в нескольких контекстах — продублировать квадрат, не тянуть стрелки
- Важные поля объекта: выписать списком или сделать отдельным квадратом (для статусов, тегов, аудитлогов)
- Указать кардинальность связей: has_one (один к одному), has_many (один ко многим)

**Правило дублирования**: «Ордер на складе» и «Ордер в магазине» — разные объекты в разных контекстах, хотя называются одинаково. Дублирование квадратов лучше нагромождения стрелок.

### Шаг 3: Найти связи и владельцев данных
- **Владелец данных (Data Owner)** — контекст, который создаёт, редактирует или удаляет данные. Определяется по командам в ES-модели.
- Стрелка направлена от владельца к потребителю (Owner → Consumer)
- Толстая стрелка: данные мутируются здесь. Тонкая стрелка: данные только читаются.
- Покрасить разделяемые данные в отдельный цвет для наглядности

**Финал**: добавить легенду (что значит каждый цвет и вид стрелки) — через месяц без неё никто не поймёт.

## Элементы нотации

| Элемент | Описание |
|---------|----------|
| Квадрат | Объект данных / контекст |
| has_one стрелка | Один к одному (у подписки — один ежемесячный пакет) |
| has_many стрелка | Один ко многим (у клиента — много подписок) |
| Жирная стрелка | Данные здесь мутируются (владелец → читатель) |
| Список полей | Важные поля объекта, специфичные для контекста |
| Цветной квадрат | Разделяемые данные или статусы |

## Примеры

**Магазин:** Client → has_many → Subscription → has_one → MonthlyPack → has_many → Item → is → Toy. Склад владеет данными об игрушке (создаёт/мутирует), Магазин только читает.

**КИБЕР-ГЕНПОДРЯД (строительство):**
- Контексты из ES: Объект, Работники, ГПР, Приёмка, Акты (КС-2/КС-3), Финансы
- Объект данных «Вид работ» нужен в ГПР, Приёмке и Актах → продублировать квадрат в каждом
- Владелец «Вид работ» → контекст «ГПР» (ИТР создаёт позиции плана)
- Владелец «КС-2» → контекст «Приёмка» (прораб создаёт акт)
- «Рабочий» нужен в ГПР (кто выполнял), Приёмке (чья подпись), Финансах (кому платить) → три квадрата «Рабочий» в трёх контекстах
- Несколько владельцев данных о рабочем → проблема: либо назначить одного владельца (HR-контекст), либо вернуться к ES и пересмотреть группировку

## Признаки проблем в модели

- **Несколько владельцев одних данных** — риск проблем с транзакционностью и консистентностью. Если возникают вопросы «как сделать сагу для двух сервисов» — точно эта проблема.
- **Слишком много стрелок между двумя контекстами** — возможно, их нужно объединить
- **Линии пересекаются неявно** — сделать явное перпендикулярное пересечение или перегруппировать

## Anti-patterns

- **Мешать концептуальную и логическую модели**: рисовать таблицы с полями вместо бизнес-объектов
- **Один объект с кучей стрелок**: дублирование квадратов понятнее нагромождения стрелок
- **Нет легенды**: через месяц никто не вспомнит, что значит пунктирная стрелка
- **Тащить все данные**: в модели только те данные, на которые завязаны бизнес-процессы
- **Создавать модель без ES**: ES и модель данных работают вместе; без ES нет контекстов для группировки данных
- **Пропустить поиск владельца данных**: если непонятно, кто владеет данными — невозможно правильно разрезать систему
