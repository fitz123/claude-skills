## Определение

**Tactical Forking** — Code First паттерн декомпозиции «монолита». Идея: скопировать существующее приложение и **отсечь лишнее** в каждой из копий, формируя новые сервисы.

Это обратный вариант [Strangler Fig](strangler-fig.md): вместо постепенного переноса функциональности в новый сервис — берёшь копию всего и удаляешь лишнее. Как создание скульптуры из куска камня.

## Когда применять

- Когда нужно вынести несколько частей из одного «монолита» одновременно
- Когда связанность между частями «монолита» слишком сложная для итеративного переноса
- Когда разные команды должны работать параллельно над разными частями системы
- Когда граница между частями понятна, и удалять код легче, чем его выносить

**Выбирай Tactical Forking, а не Strangler Fig, если:**
- Команда не хочет/не может менять стек (реализуется на том же языке)
- Нужно быстро разделить работу между командами

## Как применять

### 4 шага Tactical Forking

**Шаг 1: Копируем «монолит»**
Создаём копию (или несколько копий) распиливаемого сервиса. Каждая копия — будущий отдельный сервис.

**Шаг 2: Удаляем лишнее**
- В старой версии: удаляем код, который выносим в новый сервис
- В новой копии: удаляем код, который НЕ относится к выносимому функционалу

**Шаг 3: Настраиваем proxy-слой**
Добавляем proxy/gateway, который роутит запросы между старым и новым сервисом. Без этого нельзя переключиться между версиями.

**Шаг 4: Завершаем распил**
Когда в старом сервисе не осталось кода выносимой части, а в новом — кода старой части, распил завершён.

### Выбор между Strangler Fig и Tactical Forking

| Критерий | Strangler Fig | Tactical Forking |
|---|---|---|
| Нужен другой язык | ✅ Да (пишется на любом языке) | ❌ Нет (только тот же стек) |
| Удалять код легче, чем выносить | ❌ | ✅ Да |
| Работа нескольких команд параллельно | ❌ Сложнее | ✅ Проще |
| Скрытый неиспользуемый код | ❌ Нет | ⚠️ Остаётся |
| Качество нового кода | ✅ Чище | ⚠️ Такое же, как в монолите |

## Плюсы и минусы

**Плюсы:**
- Можно делать без предварительного глубокого анализа — команды работают независимо
- Удалять код проще, чем выносить (особенно при высокой связанности)
- Позволяет разделить работу между несвязанными командами (не пересекаются по функционалу)
- Rollback: всегда остаётся старый сервис, на который можно переключиться

**Минусы:**
- Оставляет скрытый неиспользуемый код («мёртвый код» из «монолита»)
- Качество нового кода будет таким же, как в оригинале — просто меньше его
- Создаёт несоответствия в названиях файлов и компонентов
- Реализуется только на том же технологическом стеке

## Примеры

### Happy Cat Box
**Задача:** вынести генерацию отчётов из сервиса тестирования игрушек.

Ситуация: у генерации отчётов нагрузка 1k rps, у остального тестирования — 1.5 кота в час. Разные характеристики нельзя совместить в одном сервисе.

**Решение через Tactical Forking:**
1. Копируем код сервиса тестирования игрушек → получаем 2 копии
2. В копии-для-отчётов: удаляем весь код тестирования, оставляем только генерацию отчётов
3. В оригинале: удаляем код генерации отчётов
4. Настраиваем proxy для роутинга запросов
5. Когда у оригинала нет отчётного кода, у копии — нет тестового: готово

**Альтернатива:** Strangler Fig (если хотели сменить язык).

### КИБЕР-ГЕНПОДРЯД
**Сценарий:** монолит «приёмки» включает: фиксацию факта выполнения, верификацию объёма, генерацию КС-2.

Тактический форк:
1. Копии: `acceptance-fact`, `acceptance-verification`, `ks2-generation`
2. Из `acceptance-fact` удаляем верификацию и генерацию КС-2
3. Из `acceptance-verification` удаляем фиксацию и генерацию
4. И т.д.
5. Proxy роутит по типу операции

## Anti-patterns

- **Форкнуть и не удалять лишнее** — получаем несколько одинаковых «монолитов»
- **Форкнуть без proxy** — нет возможности переключаться между версиями, нет rollback
- **Менять бизнес-логику при форке** — дополнительная сложность. Сначала форк, потом рефакторинг отдельно
- **Использовать для смены стека** — для другого языка лучше Strangler Fig
- **Форкать без анализа границ** — если границы сервисов неверны, получишь тот же каплинг, только в нескольких копиях
