## Определение

**Антипаттерн «Отдельный сервис нотификаций»** — создание выделенного технического сервиса, который получает данные и **сам решает**, как собирать темплейты и отправлять нотификации пользователям.

Проблема в том, что такой сервис неизбежно содержит бизнес-логику из других bounded context. При изменении бизнес-процессов в любом из связанных сервисов — меняются нотификации, что ломает другие процессы.

> Нотификации — это **технический шаг реализации бизнесовой команды**, а не самостоятельный bounded context.

## Почему нотификации — это не сервис

Аналогия из курса: странно делать в Event Storming отдельную бизнес-команду «отобразить пришедшую оплату на странице браузера». Нотификация — это просто способ отображения результата выполненной бизнес-команды, на который можно посмотреть, а можно и не смотреть. Единственное отличие от уведомления в браузере — нотификации могут приходить с опозданием.

## Признаки антипаттерна

- Сервис нотификаций получает «набор данных» и сам решает, что с ним делать
- Изменения в сервисе магазина → нужно менять сервис нотификаций
- Изменения в сервисе тестирования → снова нужно менять сервис нотификаций
- Разные команды разработки постоянно конфликтуют из-за сервиса нотификаций
- Instability core-сервисов = 1 (нестабильны!), а сервиса нотификаций = 0 (стабилен) — парадокс

## Расчёт instability (почему это проблема)

**С отдельным сервисом нотификаций:**
```
Каждый core-сервис:  Ca=0, Ce=1  → I = 1/(0+1) = 1.0 (максимально нестабильный!)
Сервис нотификаций:  Ca=4, Ce=0  → I = 0/(4+0) = 0.0 (стабильный)
```

Ситуация абсурдна: важные core-сервисы нестабильны (I=1), а технический — стабилен (I=0). Любое изменение логики нотификаций может сломать core-процессы.

**Без сервиса нотификаций (логика в каждом bounded context):**
```
Каждый сервис:  Ca=0, Ce=0  → I = 0/(0+0) = 0.0 (стабильный!)
```

## Как правильно

### Вариант 1: Техническая часть — технический сервис (правильный split)

Разделить нотификации на **две части**:
1. **Бизнесовая логика** (какое содержание нотификации, когда отправлять) — в каждом bounded context
2. **Техническая реализация** (протокол отправки: SMTP, APNS, FCM) — технический сервис

```
ПРАВИЛЬНО:
Магазин        → FormOrderNotification (бизнес-команда) → SendEmail (технический шаг)
Тестирование   → FormTestResultNotification             → SendEmail
```

Технический сервис отправки — допустим, потому что он реализует базовый неизменяемый функционал (взаимодействие с SMTP/IMAP протоколами). Явных связей будет больше, но скрытых — меньше, глобальная сложность ниже.

### Вариант 2: Логика нотификаций полностью в каждом bounded context

Каждый сервис сам формирует и отправляет нотификации. Нет общего сервиса нотификаций вообще.

Когда выбирать: когда кажется, что технический сервис отправки создаёт лишний каплинг (например, разные каналы отправки для разных сервисов).

## Тонкая грань: что допустимо

Нотификации имеют два шага:
1. **Создание сообщения** (содержит бизнес-логику) → всегда в bounded context!
2. **Взаимодействие с протоколом отправки** (SMTP, FCM, etc.) → можно в техническом сервисе

Если объединить логику (шаг 1) в общий сервис → высокая **скрытая** связанность, глобальная сложность растёт.

Если объединить только протокол (шаг 2) → **явная** связанность, контролируемая, скрытой нет.

## Примеры

### Happy Cat Box
**Проблема:** сервис нотификаций получал данные и сам решал, какой темплейт выбрать. В нём срослась логика из магазина и тестирования. Итог: две команды постоянно ругались и случайно ломали друг друга.

**Решение Ибрагима:** удалить сервис нотификаций, перенести логику нотификаций обратно в каждый bounded context.

### КИБЕР-ГЕНПОДРЯД
**Неправильно:** создать «сервис уведомлений строительства», который сам решает:
- рабочий получает уведомление о новом задании
- прораб получает уведомление о закрытии наряда
- РПП получает уведомление о готовности к приёмке

Каждый бизнес-процесс будет тянуть сервис в свою сторону.

**Правильно:** каждый модуль сам отправляет нотификации:
- Модуль заданий → уведомление рабочему при назначении
- Модуль приёмки → уведомление прорабу при закрытии наряда

## Anti-patterns

- **«Нотификации — отдельный поддомен»** — нет, это технический шаг
- **Общие темплейты для разных bounded context** — скрытый каплинг, который взорвётся при первом изменении
- **Сервис нотификаций как «простой первый шаг» при распиле монолита** — добавляет каплинг, не приносит пользы
- **Логика «когда отправлять» в сервисе нотификаций** — это бизнес-логика, она принадлежит bounded context
