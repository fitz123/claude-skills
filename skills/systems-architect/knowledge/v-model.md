## Определение

V-Model — методологическая модель разработки, разработанная независимо в США и Германии в конце 80-х. Делит процесс на три части по временной шкале: **дефинишен** (определение) → **реализация** → **тестирование**. Ключевая идея: каждому этапу дефинишена соответствует свой уровень тестирования. Цикл повторяется если тест провалится.

**Архитектура в V-Model**: архитектор описывает структуру и ограничения (стратегия), а дизайн — детали реализации (тактика) — делегируется команде. Это предотвращает ситуацию, когда разработчики переделывают систему «по-своему».

## Когда применять

- При разработке нового продукта или системы
- При определении, что архитектор описывает команде, а что остаётся на усмотрение разработчиков
- При выборе уровня тестирования для проверки архитектурных решений
- При объяснении команде, почему нужно сначала описать структуру до кодирования

## Как применять

**Этапы V-Model:**

**Левая ветка (Дефинишен):**
1. **User requirements** — что хочет бизнес (функциональные требования, стейкхолдеры)
2. **Architecture** — архитектурный стиль, границы контекстов, характеристики (архитектура)
3. **Design** — паттерны, БД, библиотеки, детали реализации (тактика команды)
4. **Implementation** — код

**Правая ветка (Тестирование, зеркально к Дефинишену):**
1. **Unit Testing** ↔ Design
2. **Integration Testing** ↔ Architecture
3. **System Verification & Validation** ↔ User Requirements

**Правило работы:**
- Если уровень тестирования провален → возврат к связанному этапу левой ветки
- После возврата повторяются все последующие шаги: дизайн → реализация → тесты
- Чем выше уровень провала, тем дороже возврат

**Разделение архитектуры и дизайна:**
- **Архитектура** (задача архитектора): структура системы, границы, виды коммуникаций, характеристики, ограничения
- **Дизайн** (задача команды): конкретные технологии, паттерны, схемы БД, структура кода

**Правило**: архитектор описывает *что должно быть* и задаёт ограничения → команда решает *как сделать* в рамках ограничений.

## Примеры

**Правильное использование:**
- Архитектор описывает: «Каждый боундед-контекст имеет изолированную БД, коммуникация через async events»
- Команда решает: PostgreSQL vs MongoDB, RabbitMQ vs Kafka, конкретные схемы таблиц

**Ошибка Ибрагима (кейс из курса):**
- Ибрагим описал детальную реализацию системы для разработчиков (дизайн вместо архитектуры)
- Разработчики «переделали по-своему» → монолит 1M строк → система стала неуправляемой
- Правильно: описать структуру и ограничения, делегировать дизайн

**Применительно к КИБЕР-ГЕНПОДРЯД:**
- Архитектор задаёт: «Модуль ГПР и модуль Приёмки изолированы, интеграция через события "Задание принято"»
- Команда решает: как хранить плановые объёмы, какой ORM использовать, структуру API

## Anti-patterns

- **Пропустить дефинишен**: начать кодить без описания архитектуры → команда принимает стратегические решения сама → хаос
- **Архитектор описывает дизайн**: диктовать названия классов, схемы БД → убивает инициативу, разработчики всё равно переделают
- **Воспринимать как строго линейный процесс**: V-model цикличен, итерации нормальны
- **Не проводить валидацию на каждом уровне**: пропустить System Verification → дорогой возврат к требованиям
- **Архитектура без ограничений**: описать «сделайте микросервисы» без правил → распределённый монолит
