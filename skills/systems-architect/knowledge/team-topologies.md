## Определение

Team Topologies — подход, в котором архитектура системы строится вокруг структуры команд, а не только вокруг бизнес-проблем. Фактически формализованный **Закон Конвея**: «системы проектируются так, что их структура копирует коммуникационную структуру организации».

Авторы: Matthew Skelton и Manuel Pais, книга «Team Topologies» (2019).

**Ключевая идея**: есть две роли команд:
1. **Те, кто работает** — создают ценность для пользователя
2. **Те, кто помогает работать лучше** — снижают когнитивную нагрузку первых

## Когда применять

- При масштабировании команды (>10-15 человек)
- При разбиении монолита, когда непонятно как делить команды
- Когда команды мешают друг другу и много coordination overhead
- При проектировании новой структуры компании или реорганизации
- В связке с Wardley Maps для выравнивания структуры системы и команд

## Как применять

### 4 типа команд

| Тип | Роль | Пример |
|-----|------|--------|
| **Stream-aligned** | Работают над потоком ценности, владеют продуктом/фичей | Команда «Приёмка работ» |
| **Enabling** | Помогают stream-aligned освоить новые практики | DevOps-гильдия, архитектурный совет |
| **Platform** | Предоставляют внутреннюю платформу (API, инфра) | Команда внутренней платформы |
| **Complicated Subsystem** | Работают с высокосложными частями, требующими экспертизы | ML-команда, cryptography |

### 3 способа взаимодействия команд

| Способ | Описание | Когда |
|--------|----------|-------|
| **Collaboration** | Тесная совместная работа | Исследование, новые домены (временно) |
| **X-as-a-Service** | Одна команда потребляет сервис другой | Платформенные зависимости |
| **Facilitating** | Enabling-команда помогает освоить практику | Обучение, онбординг |

### Алгоритм применения

1. Определить потоки ценности в бизнесе (stream = путь фичи от идеи до пользователя)
2. Сформировать stream-aligned команды вокруг этих потоков
3. Выявить когнитивные перегрузки: где команды тонут?
4. Добавить enabling- и platform-команды для снятия перегрузки
5. Убедиться, что границы команд совпадают с границами bounded contexts
6. Контролировать способы взаимодействия — collaboration должно быть временным

### Связь с архитектурой

- **Монолит** → команды коммуницируют больше, выше coupling между людьми
- **Микросервисы** → команды коммуницируют меньше, выше автономия
- Архитектурный стиль должен соответствовать структуре команд (иначе Закон Конвея сломает всё)

## Примеры

### КИБЕР-ГЕНПОДРЯД (стартап, ~3-5 чел.)

На стартапе тяжело применять полные Team Topologies — нет масштаба. Но принципы работают:

**Stream-aligned команды** (один человек может вести несколько):
- Stream «Производство»: ГПР + приёмка работ
- Stream «Финансы»: процентовка, КС-2/КС-3

**Enabling** (роль, не отдельный человек):
- Архитектурные ревью и решения — техлид (Ninja)

**X-as-a-Service:**
- Auth, уведомления — внешние сервисы (Firebase, SMS-провайдер)

**Вывод**: при росте до 10+ человек — делить команды по потокам ценности, а не по техническим слоям (не «фронт-команда» и «бэк-команда»).

## Anti-patterns

- **Команды по технологиям**: «frontend team» + «backend team» + «DBA team» → все зависят от всех, ни один поток не автономен
- **Вечный Collaboration**: collaboration должен быть временным; если постоянный — это сигнал неправильных границ
- **Игнорировать когнитивную нагрузку**: команда не должна знать слишком много, чтобы сделать одну фичу
- **Копировать org-chart в архитектуру**: оргструктура не всегда оптимальна; иногда нужно наоборот — менять org-chart под желаемую архитектуру
