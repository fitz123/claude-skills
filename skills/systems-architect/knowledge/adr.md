## Определение

ADR (Architecture Decision Record) — короткий текстовый документ (1-2 страницы), описывающий конкретное архитектурное или дизайнерское решение вместе с контекстом его принятия. Главная ценность — сохранение контекста: почему было выбрано именно это решение, какие альтернативы рассматривались, при каких условиях. Без ADR новые разработчики либо повторяют прошлые ошибки, либо меняют решения без понимания причин.

**ADR ≠ описание архитектуры**: ADR описывает *конкретное решение*, а не всю систему. Для описания архитектуры — C4, arc42 (см. architecture-frameworks.md).

## Когда применять

- При любом нетривиальном архитектурном решении (выбор стиля, БД, коммуникации, паттерна)
- Когда решение влияет на несколько команд
- При выборе между двумя и более равнозначными альтернативами
- Когда решение не очевидно и через год его причина будет непонятна

## Как применять

**Обязательные поля ADR:**

```markdown
# ADR-[номер]: [Короткое описание решения]

## Status
[Proposed | Accepted | Rejected | Superseded by ADR-XXX]

## Context
Описание ситуации и проблемы, которую нужно решить. Почему вообще
принимается это решение? Какие ограничения и характеристики влияют?
Описать альтернативы можно здесь или в отдельной секции Alternatives.

## Decision
Описание принятого решения. Акцент на **ПОЧЕМУ**, не на КАК.
Используйте директивный язык: «мы будем использовать...»,
«система должна...» — не «нам кажется, что...».

## Consequences
Влияние решения на систему. Trade-offs: что улучшается, что ухудшается.
```

**Дополнительные поля:**

```markdown
## Compliance
Как проверить соблюдение решения: автоматически (fitness function)
или вручную.

## Notes
Метаинформация: автор, дата, кто апрувил, история изменений.

## Alternatives
Список рассмотренных альтернатив с анализом плюсов/минусов.
```

**Статусы ADR:**
- **Proposed** — ждёт апрува (архитектурного комитета, тимлида)
- **Accepted** — принято, идёт в реализацию
- **Rejected** — отклонено с описанием причины (важно сохранить!)
- **Superseded by ADR-N** — заменено другим решением (сохраняет исторический контекст)

**Когда нужен апрув других:**
- Высокая цена решения → апруввать с менеджером или лидом
- Решение влияет на несколько команд → апрувить с командами
- Связано с security → апрувить с security-командой

**Хранение ADR:**
- Каждый ADR — отдельный файл, нумерованный (ADR-001, ADR-002)
- Лучше в git-репозитории или Confluence
- Структура папок: `adr/common/`, `adr/service-name/`, `adr/integration/`, `adr/enterprise/`

## Примеры

**ADR-001: Использование микросервисной архитектуры**
```
Status: Accepted

Context: Система Happy Cat Box требует независимого деплоя для сервиса 
тестирования (релиз раз в день) vs. остальные сервисы (раз в 2 недели).
Финансовый комплаенс требует иммутабельной БД только для биллинга.
Объединить эти характеристики в монолите невозможно без распространения
ограничений на всю систему.

Decision: Мы будем использовать микросервисную архитектуру с отдельными
БД для каждого сервиса. Каждый сервис изолирует свои характеристики
и ограничения.

Consequences: (+) Независимый деплой. (+) Изоляция ограничений.
(-) Высокая операционная сложность. (-) Нужна инфраструктура для
межсервисного взаимодействия и наблюдаемости.
```

**КИБЕР-ГЕНПОДРЯД — пример ADR:**
```
ADR-003: Offline-first для мобильного клиента рабочих

Context: Стройплощадки часто не имеют стабильного интернета.
Рабочим нужно фиксировать выполненные работы в реальном времени.
Потеря данных недопустима (влияет на ЗП и КС-2).

Decision: Мы будем использовать offline-first архитектуру для мобильного
клиента с локальной SQLite и синхронизацией при появлении сети.

Consequences: (+) Работа без сети. (+) Нет потери данных.
(-) Конфликты синхронизации нужно разрешать. (-) Сложнее тестировать.
```

## Anti-patterns

- **Один ADR на несколько решений**: «выбор стиля и БД» — это N+1 решений, не одно
- **Описывать КАК, не ПОЧЕМУ**: технические детали реализации — не ADR, а документация кода
- **Не описывать Superseded-ADR**: потеря контекста — «почему мы перешли от монолита?»
- **Пассивный язык**: «нам кажется, что лучше использовать...» → «мы будем использовать»
- **ADR без апрува для важных решений**: критичные решения без согласования → сюрпризы при реализации
- **Не хранить Rejected-ADR**: будущие разработчики будут снова предлагать те же идеи
