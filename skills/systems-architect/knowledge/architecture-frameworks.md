## Определение

Архитектурные фреймворки — стандартизированные способы описания архитектуры системы для разных аудиторий. ADR описывает конкретные решения, но не архитектуру целиком. Для полного описания используют: C4, 4+1, arc42, ISO 42010.

## Когда применять

- При создании документации для новой системы
- При онбординге новых разработчиков
- При коммуникации архитектуры с разными стейкхолдерами (бизнес vs техническая команда)
- При аудите или передаче системы

## Как применять

**C4 Model** (4 уровня детализации):
1. **Context** (L1) — система в окружении: акторы + внешние системы. Для всех стейкхолдеров
2. **Container** (L2) — внутри системы: сервисы, приложения, БД. Для технических руководителей
3. **Component** (L3) — внутри контейнера: компоненты и их зависимости. Для разработчиков
4. **Code** (L4) — детали реализации: классы, функции. Редко, только критические части

**4+1 Views**: Logical, Process, Development, Physical + Scenarios (use cases)

**arc42**: 12-секционный шаблон документа: цели → ограничения → контекст → решения → etc.

**Для веб-проектов достаточно**: C4 L1-L2 + sequence diagrams + характеристики + ADR

## Примеры

- C4 L1: пользователь → [наш сервис] → [payment gateway] (3 прямоугольника и 2 стрелки)
- C4 L2: frontend → backend API → БД + message broker + cache
- arc42 используется в enterprise с формальными требованиями к документации

## С чего начать изучение фреймворков

Если не хочется погружаться в каждый фреймворк сразу — **начните с arc42**: он самый популярный в индустрии и наименее «душный» из всех. После освоения arc42 легче понять остальные.

Порядок изучения:
1. **arc42** — базовый шаблон, хорош для старта
2. **C4** — лучший для визуализации, особенно в инженерных командах
3. **4+1** — если нужно описывать разные viewpoints для разных аудиторий
4. **ISO 42010** — если работаете в enterprise с формальными требованиями к AD

## Anti-patterns

- **Документировать только L3/L4**: детали без контекста — никто не понимает зачем
- **Только текстовое описание**: без диаграмм архитектура непонятна большинству
- **Использовать arc42 для стартапа в production**: overhead перевесит пользу, но для изучения — подходит
- **Не обновлять после изменений**: устаревшая документация хуже её отсутствия
- **Придумывать свой шаблон**: фреймворки существуют чтобы не думать «что описывать» — использовать готовые
