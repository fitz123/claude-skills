## Определение

Поддомен — составная часть бизнес-домена, описывающая конкретную бизнес-проблему или бизнес-функцию. Поддомены существуют в **Problem Space** и являются строительными блоками для определения боундед-контекстов. Каждый поддомен классифицируется как Core, Generic или Supporting.

## Когда применять

- При декомпозиции бизнес-домена на части
- При расстановке приоритетов для команды
- При решении о buy vs build
- При распределении команд по областям ответственности

## Как применять

**Найти поддомены:**
1. Начать с департаментов компании как стартовой точки
2. Внутри каждого департамента найти отдельные бизнес-функции
3. Проверить: каждый поддомен = отдельная проблема, не техническое решение

**Классифицировать поддомены (быстрый тест):**
1. Можно купить готовое решение и компания останется собой? → **Generic** (купить/аутсорс)
2. Справятся ли джуны без контроля качества? → **Supporting** (инхаус или аутсорс, низкий приоритет)
3. Нет к обоим? → **Core** (инхаус, лучшие люди, максимальный приоритет)

| | Core | Generic | Supporting |
|---|---|---|---|
| Конкурентное преимущество | Да | Нет | Нет |
| Сложность модели | Высокая | Любая | Низкая |
| Изменчивость | Частая | Редкая | Редкая |
| Команда | Лучшие | Купить/аутсорс | Джуны/аутсорс |

## Примеры

- Маркетплейс: Core = алгоритм ранжирования товаров; Generic = авторизация пользователей (Auth0); Supporting = генерация PDF-чеков
- Банк: Core = скоринг кредитных рисков; Generic = SMS-нотификации; Supporting = внутренний тикет-трекер

## Anti-patterns

- **Всё Core**: переоценивать уникальность своих решений → тратить дорогих людей на generic задачи
- **Следовать оргструктуре**: 1 отдел = 1 поддомен — один отдел может решать несколько задач
- **Технические поддомены**: «поддомен авторизации» — это generic, не отдельный бизнес-поддомен
- **Не пересматривать**: то что было Core 2 года назад, может стать Generic (рынок дозрел)
