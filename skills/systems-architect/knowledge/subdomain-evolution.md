## Определение

Поддомены не статичны — они эволюционируют вместе с бизнесом. То, что сегодня является generic (покупным) решением, завтра может стать core (уникальным конкурентным преимуществом). И наоборот — core-поддомен может упроститься до supporting или commoditize до generic.

**Пример трансформации:**
- YouTube начинал как дейтинг-приложение → стал сервисом видео
- Happy Cat Box: склад был generic (стандартное решение) → стал core (SaaS, продаваемый другим бизнесам)

> Это значит, что при принятии решений нужно учитывать, что расклад может поменяться, и продумывать дополнительные пути развития.

## Типы трансформаций

```
Core → Supporting:  Упрощение бизнес-модели, конкурентное преимущество исчезло
Core → Generic:     Функция стала стандартом отрасли (commoditization)
Generic → Core:     Компания нашла в generic-решении уникальную ценность
Generic → Supporting: Решение перестало быть покупным, стало нужным, но не ключевым
Supporting → Core:  Supporting-функция стала стратегически важной
```

**Схема трансформаций (из Vladik Khononov):**
- Core упрощается → Supporting
- Core становится стандартом → Generic
- Generic получает уникальность → Core

## Почему это важно

При смене поддомена меняются **характеристики** всех bounded context в нём. Это влечёт:

1. Изменение архитектурного стиля (layered → microkernel)
2. Изменение требований к БД
3. Изменение способов коммуникации
4. Изменение бюджета на разработку (core = больше инвестиций)
5. Необходимость рефакторинга существующих сервисов

**Пример:** склад Happy Cat Box стал продаваться как SaaS:
- Было: generic, характеристики простые (layered)
- Стало: core, нужны modifiability + maintainability → нужен microkernel

## Как отслеживать: Core Domain Charts

**Core Domain Chart** — диаграмма, показывающая текущее состояние всех поддоменов с прогнозом их эволюции.

### Как использовать

1. Нарисуй все поддомены как точки на оси «уникальность × сложность»
2. Добавь стрелки: куда и почему будет мигрировать каждый поддомен
3. Обновляй регулярно при изменении стратегии бизнеса

```
Ось Y: Уникальность (насколько это конкурентное преимущество)
Ось X: Сложность модели

Core:       высокая уникальность, высокая сложность
Supporting: средняя уникальность, средняя сложность  
Generic:    низкая уникальность (покупное), любая сложность
```

### Зачем прогнозировать трансформации

1. **Технический рефакторинг**: заранее подготовить систему к новым характеристикам
2. **Продажа рефакторинга бизнесу**: «Этот поддомен скоро станет core, нам нужно переписать его заранее»
3. **Планирование инфраструктуры**: core требует другой уровень надёжности и инвестиций

## Алгоритм действий при трансформации

### Шаг 1: Зафиксировать факт трансформации

Признаки, что поддомен трансформируется:
- Бизнес начал активно инвестировать в функцию
- Функция стала продаваться внешним клиентам
- Конкуренты начали копировать эту функцию
- Функция перестала быть дифференциатором (commoditization)

### Шаг 2: Определить новые характеристики

| Тип | Ключевые характеристики |
|---|---|
| Generic → Core | +modifiability, +maintainability, +evolvability, +budget |
| Core → Generic | -modifiability, +simplicity, можно купить готовое решение |

### Шаг 3: Выбрать стратегию адаптации

- Изменить архитектурный стиль bounded context
- Пересмотреть выбор БД
- Изменить способы коммуникации
- Написать ADR с обоснованием

### Шаг 4: Продать изменения бизнесу

Core Domain Charts — отличный инструмент для объяснения технического долга бизнесу: «Этот модуль стал нашим конкурентным преимуществом, поэтому его нужно переписать, чтобы мы могли его развивать».

## Примеры

### Happy Cat Box — склад Generic → Core

**Было:** склад как generic поддомен
- Архитектурный стиль: layered
- Характеристики: простые, без modifiability
- Бюджет: минимальный

**Событие:** бизнес начал продавать решение склада другим компаниям как SaaS

**Стало:** склад как core поддомен
- Новые характеристики: modifiability, maintainability
- Нужен новый архитектурный стиль: microkernel
- Бюджет: «сколько понадобится»
- Все коммуникации переведены на асинхронные

**Решение Ибрагима:** написать новый сервис склада в стиле microkernel, перегнать данные через CDC.

### КИБЕР-ГЕНПОДРЯД

**Потенциальные трансформации:**

| Поддомен | Сейчас | Может стать | Триггер |
|---|---|---|---|
| ГПР (планирование) | Supporting | Core | Если ГПР станет продаваться другим подрядчикам |
| Документооборот (КС-2/КС-3) | Generic | Supporting | Если нужна кастомная интеграция с заказчиком |
| Приёмка работ | Core | — | Уже core, конкурентное преимущество |
| Расчёт процентовки | Supporting | Core | Если добавить AI-оптимизацию |

## Anti-patterns

- **Игнорировать трансформации** — «у нас generic, зачем его развивать» — потом будут большие проблемы с рефакторингом
- **Реагировать постфактум** — рефакторинг core-поддомена, который уже в продакшене с высокой нагрузкой, намного дороже
- **Не документировать трансформации** — через год никто не вспомнит, почему склад вдруг стал core (ADR!)
- **Застрять в «это generic»** — если конкуренты не могут повторить — это core, неважно как называть
