## Определение

Event Storming (ES) — воркшоп для коллективного описания системы через события и процессы. Цель: выгрузить знания участников в единую модель, с которой согласна вся команда. ES отвечает на три вопроса:
1. Как ведёт себя система для разных пользователей?
2. Из каких процессов состоит система?
3. Как процессы связаны между собой?

ES — не технический воркшоп. Здесь не говорят про классы, адаптеры и базы данных. Только бизнес-поведение.

## 6 типов стикеров

| Цвет | Тип | Правило |
|------|-----|---------|
| 🟠 Оранжевый | **Событие (Event)** | Результат команды, прошедшее время, необратимо. «Заказ оплачен», не «Оплатить заказ». |
| 🔵 Голубой | **Команда (Command)** | Действие системы. «Что» происходит, не «как». «Checkout», не «Store user record». |
| 🟡 Жёлтый | **Актор (Actor)** | Кто вызывает команду: человек (менеджер, клиент) или система (мобильное приложение, умный чайник). |
| 🟣 Фиолетовый | **Полиси (Policy)** | Условие, при котором система автоматически запускает команду после события. Содержит бизнес-контекст, не «данные валидны». |
| 🩷 Розовый | **Внешняя система (External System)** | Другие системы (эквайринг, курьерская служба, scheduler/cron). Всегда возвращает событие. |
| 🟢 Зелёный | **Read Model** | Данные, необходимые актору для принятия решения о вызове команды. Собирается из событий или локальных данных. |

## Когда применять

- При старте нового проекта: понять, из каких процессов состоит система
- При анализе legacy-системы: выявить скрытые связи между модулями
- При разбиении монолита: найти правильные границы контекстов
- При онбординге: быстро синхронизировать команду о том, как работает система

## Как применять — 3 шага

### Шаг 1: Найти все события
1. Записать все события в хронологическом порядке (слева — раньше, справа — позже)
2. События, которые вызывают следующее действие — соединить стрелками
3. Добавить акторов и комментарии для контекста
4. Не нужно выписывать ВСЕ события — только базовые, запускающие следующий шаг

**Различие:** «Событие вызывает другое» (триггер) ≠ «Событие происходит во времени после» (последовательность). Оплата подписки не приводит к автоматической отмене подписки.

### Шаг 2: Добавить команды, акторов и полиси
1. Сгруппировать события по командам, которые их вызывают
2. К каждой команде добавить актора или указать событие-триггер
3. Добавить полиси где нужны условия запуска
4. Периодические действия (cron) — внешняя система "Scheduler" генерирует событие

**Правила полиси:** содержит бизнес-контекст («если на складе < N товаров — заказать у поставщика»), не технические детали.

### Шаг 3: Группировка в контексты и read models
1. Объединить команды и события в группы по логике процесса
2. Определить read models — данные, без которых актор не может принять решение
3. Очевидные read models можно пропустить, чтобы не перегружать модель
4. Указать стрелкой, из каких событий собирается read model (если это неочевидно)

## Правила именования

- **Событие**: глагол в прошедшем времени, бизнес-термин. «Пользователь зарегистрирован», не «User Created In DB»
- **Команда**: настоящее время, отвечает на «что», не «как». «Оформить заказ», не «Validate params, store record, produce event»
- **Полиси**: бизнес-условие с контекстом, не техническое
- Если в ES много стрелочно-связанных команд (3+) → вероятно, это технические шаги одной бизнес-команды

## Примеры

**Магазин:**
- Событие: «Заказ оплачен» → Команда: «Передать в сборку» → Событие: «Заказ собран» → Внешняя система: «Курьерская служба»
- Read model для актора «Сборщик»: список товаров в заказе

**КИБЕР-ГЕНПОДРЯД (строительство):**
- Событие: «Акт КС-2 подписан» → Полиси: «Если все виды работ закрыты» → Команда: «Сформировать КС-3»
- Актор: «Прораб» → Команда: «Закрыть позицию ГПР» → Событие: «Позиция ГПР закрыта»
- Read model для ИТР: текущий статус работ на объекте
- Внешняя система: «Scheduler» генерирует событие «Конец отчётного периода» → Команда «Рассчитать процентовку»
- Внешняя система: «1С» ← Команда «Передать данные для выплаты»

## Уровни воркшопа

1. **Big Picture** — разбор всех процессов в системе целиком
2. **Process Modeling** — детальный разбор одного процесса
3. **Software Design** — маппинг domain model patterns на ES

## Anti-patterns

- **Технические события**: «User Created In DB», «Form Submitted», CRUD-операции — это не бизнес-события
- **Технические шаги вместо команды**: если 3+ события связаны стрелками подряд — это один бизнес-процесс, не несколько
- **Полиси без контекста**: «если событие получено», «если данные валидны» — бессмысленно
- **Уйти в технические детали**: начать обсуждать, как писать код, какую БД выбрать
- **Нотификации как отдельный контекст**: нотификации — техническая команда, не бизнес-процесс. Если они вызываются после каждого действия → технический шаг, не отдельный bounded context
