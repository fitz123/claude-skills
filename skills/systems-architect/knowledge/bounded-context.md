## Определение

Bounded Context — изолированная область системы, в которой Ubiquitous Language однозначен. Каждый контекст решает конкретную бизнес-проблему и имеет чёткие границы. Это единица в **Solution Space** — ответ на проблему из Problem Space.

## Когда применять

- При декомпозиции системы на независимые части
- При определении владения кодом командами
- При проектировании микросервисов и модулей
- При анализе ES-модели: группировать связанные события в контексты

## Как применять

1. **Начать с поддоменов**: каждый поддомен → кандидат в контекст
2. **Проверить правила**:
   - 1 поддомен ≈ 1 контекст (стремиться, не догма)
   - 1 контекст = 1 команда разработки (обязательно)
   - 1 команда может владеть несколькими контекстами
3. **Провести Event Storming**: сгруппировать события по бизнес-процессам → увидеть контексты
4. **Построить модель данных**: много связей между двумя блоками → возможно один контекст
5. **Проверить размер**: много поддоменов → слишком большой контекст, дели; много контекстов на один поддомен → вероятно перерезали

## Примеры

- Электронная торговля: Checkout, Inventory, Catalog, Shipping, Billing — отдельные контексты
- «Биллинг» — часто 2 контекста: «Оплата клиентом» и «Выплата подрядчику» — разные процессы и команды
- Чекаут ≠ Корзина + Оплата — нельзя резать единый бизнес-процесс на контексты без причины

## Anti-patterns

- **Технические названия**: «Сервис уведомлений» — это не контекст, это техническая деталь
- **Один контекст на несколько поддоменов**: слишком большой, нет изоляции
- **Резать единый процесс**: Чекаут = Корзина + Оплата → неоправданное усложнение
- **Искать «идеальные» границы в одиночку**: нужен фидбэк команды и domain expert'ов
- **Entity service как контекст**: «UserContext», «OrderContext» — строить вокруг сущностей, не процессов
