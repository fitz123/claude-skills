## Определение

Microkernel Architecture (Plugin Architecture) — монолитный стиль, состоящий из минимального ядра с базовой функциональностью и набора подключаемых плагинов. Ядро определяет интерфейс плагина; плагины реализуют изменяемую логику независимо друг от друга.

## Когда применять

- Логика часто меняется и разнообразна (алгоритмы обработки, бизнес-правила)
- Нужна расширяемость без изменения ядра
- Разные клиенты нуждаются в разных «версиях» поведения
- IDE, плагинные системы, ETL-пайплайны, rule engines

## Как применять

1. **Выделить стабильное ядро**: минимальный набор функций, который не меняется
2. **Определить интерфейс плагина**: контракт, которому должен следовать каждый плагин
3. **Реализовать плагины**: каждый плагин = одна изменяемая единица поведения
4. **Ядро загружает плагины**: через registry, DI-контейнер, файловую систему
5. **Плагины не знают друг о друге**: нет зависимостей между плагинами

## Примеры

- **IDE**: ядро = редактор + навигация; плагины = поддержка языков, линтеры, дебаггеры
- **ETL-система**: ядро = планировщик и оркестратор; плагины = коннекторы к разным источникам
- **Rule Engine**: ядро = движок применения правил; плагины = конкретные бизнес-правила (скидки, фроды)
- Браузер: ядро = рендеринг; плагины = расширения

## Anti-patterns

- **Сложная логика в ядре**: ядро должно быть минимальным — всё в плагины
- **Зависимости между плагинами**: плагины должны быть независимы
- **Использовать там где логика стабильна**: overhead без выгоды; лучше Layered или Modular Monolith
- **Нет чёткого интерфейса плагина**: каждый плагин делает что хочет → хаос
