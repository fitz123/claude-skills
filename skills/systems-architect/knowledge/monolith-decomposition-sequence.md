## Определение

Процесс распила монолита на сервисы требует **чёткого порядка действий**. Неправильная последовательность приводит к дублированию работы, потере данных, конфликтам команд и срыву сроков.

Ключевой принцип: **никогда не пилить ради самого распила**. Распил только ради улучшения конкретных характеристик.

## 8 шагов любого распила монолита

1. **Локализация проблемы** — найти, что именно не удовлетворяет характеристикам
2. **Анализ системы** — ES, data model, instability, coupling
3. **Принятие архитектурных решений** — выбор паттерна, варианта, документация в ADR
4. **Приоритизация** — в каком порядке выносить сервисы (самое сложное!)
5. **Реализация** — итеративная, с сохранением рабочего варианта
6. **Валидация** — Shadow Mode, параллельный запуск, сравнение результатов
7. **Вывод в эксплуатацию** — постепенное переключение трафика
8. **Закрытие технического долга** — удаление старого кода

## Как приоритизировать: 2 ситуации

### Ситуация 1: Есть ресурсы, нет опыта и/или инфраструктуры

**Стратегия: Начинать с безопасного для обучения**

Цель — набрать опыт и проверить инфраструктуру на некритичных сервисах, прежде чем трогать важное.

**Порядок:**

1. **Добавить новую функциональность как отдельный сервис** — самый безопасный старт. Нет старого кода, нет рисков сломать. Идеально для проверки инфраструктуры.

2. **Вынести наименее рисковый existing сервис** — generic или support поддомен. В Happy Cat Box — сервис инфлюенсеров. Если сломается — core-бизнес продолжает работать.

3. **Двигаться от менее к более критичным** — после получения опыта переходить к более важным core-сервисам.

4. **Core-поддомены — в последнюю очередь** — только когда команда уже умеет распиливать.

**Пример для Happy Cat Box (нет опыта):**
```
1. Новый функционал: расчёт зарплат склада (изолированный, не ломает монолит)
2. Сервис инфлюенсеров (support, безопасный для экспериментов)  
3. Расчёт зарплат тестировщиков (core, но однонаправленные коммуникации)
4. Генерация отчётов (core, небольшой кусок)
5. Магазин (core, важный, нагрузка)
6. Расчёт параметров игрушек (core, много консёрнов)
7. Тестирование игрушек (core, нужна смена стиля)
8. Склад + generic-поддомены
```

---

### Ситуация 2: Есть опыт и/или инфраструктура, нет ресурсов

**Стратегия: Начинать с максимального эффекта при минимальных усилиях**

Цель — быстро улучшить самые важные части системы, которые приносят деньги или нарушают compliance.

**Критерии приоритизации:**
- Core-поддомены, которые не удовлетворяют характеристикам → первые
- Compliance/финансовые требования → высокий приоритет
- Высокая нагрузка на core → критично
- Generic и support → в конце

**Пример для Happy Cat Box (есть опыт):**
```
1. Магазин — core, приносит деньги, нагрузка рушит всё в праздники
2. Расчёт зарплат — compliance с финрегулятором, риски для компании
3. Генерация отчётов — сильно грузит систему
4. Расчёт параметров игрушек — много стейкхолдерских консёрнов
5. Тестирование игрушек — последний core-элемент
6. Работа с инфлюенсерами (support)
7. Склад и generic-поддомены
```

## Правила для обоих вариантов

### Начинай с интеграций и переноса данных

> «Всегда начинайте с интеграций и переноса данных, потому что это самая рискованная и опасная часть любого распила»

Классическая ошибка: заниматься кодом, а про интеграцию вспомнить за 2 дня до дедлайна.

### Всегда сохраняй rollback-стратегию

Все три паттерна (Strangler Fig, Tactical Forking, CDC) дают возможность держать две версии системы. Используй это:
- Proxy/gateway роутит между старым и новым
- Shadow Mode: запускай оба, сравнивай результаты
- Переключение: постепенно, не сразу весь трафик

### Не «изолировать в монолите сначала»

Подход «сначала изолируем код в монолите, потом выносим» — долгострой. Если правильно выбраны границы — быстрее сделать сервис «сбоку».

### Опиши 5 стандартов компании

Перед распилом (особенно при наличии нескольких команд) зафиксируй:
1. **Документация** — какие модели нужны (ES, data model, ADR)
2. **Коммуникации** — допустимые варианты sync/async
3. **Технологии** — список разрешённых БД, фреймворков (Tech Radar)
4. **Болванка сервиса** — шаблон для каждого языка
5. **Fitness functions** — обязательный набор метрик + инструменты

### Используй DAG для планирования

Для отслеживания прогресса нескольких параллельных потоков работы используй **DAG (Directed Acyclic Graph)** — ориентированный граф без циклов:
- Слева — задачи, которые нужно сделать раньше
- Стрелки — зависимости (нельзя начать правую, пока не сделана левая)
- Зависимости: many_to_one, one_to_one, one_to_many
- Диаграмма **неизменяема** после утверждения, только ревьюится еженедельно

## Применение в КИБЕР-ГЕНПОДРЯД

### Если нет опыта распиливания

```
1. Новая функциональность: AI-анализ ГПР (отдельный сервис, не трогает монолит)
2. Документооборот (generic, безопасный для экспериментов)
3. Расчёт зарплат рабочих (однонаправленные связи)
4. Модуль приёмки - репорты (небольшой кусок core)
5. Основной модуль приёмки (core, critical)
6. ГПР (core, основной процесс)
7. Процентовка и КС-2/КС-3
```

### Если есть опыт

```
1. Процентовка КС-2/КС-3 — compliance с ФЗ-44, финансовые риски
2. Приёмка работ — core, основной бизнес-процесс
3. ГПР — core, высокая нагрузка в пиковые периоды
4. Расчёт зарплат — compliance
5. Документооборот (generic)
```

## Anti-patterns

- **«Сначала нотификации»** — популярный первый шаг, добавляет каплинг, не улучшает характеристики
- **Распил без fitness functions** — нет способа проверить, что после распила стало лучше
- **Распил без зафиксированных стандартов** — разные команды делают по-разному, потом не интегрировать
- **Игнорировать приоритеты бизнеса** — технически правильный порядок может не совпадать с бизнес-приоритетами
- **Распиливать без rollback** — нет возможности отката = огромный риск
- **Начинать с самого большого и важного** — слишком рискованно без опыта команды
