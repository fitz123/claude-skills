## Определение

Event Sourcing — паттерн хранения данных, где состояние системы определяется не текущими значениями в таблицах, а последовательностью событий. Текущее состояние = результат применения всех событий. События иммутабельны — только добавляются, никогда не изменяются.

## Когда применять

- Нужна полная история изменений (audit log)
- Нужна возможность «отмотать» состояние на любой момент времени
- Финансовые транзакции (каждая операция — событие)
- Интеграция с CQRS (события → разные проекции)
- Сложные бизнес-процессы с reversal (отмена заказа должна создать новые события)

## Как применять

1. **Определить события домена**: OrderPlaced, PaymentProcessed, OrderShipped
2. **Создать Event Store**: иммутабельное хранилище событий (Kafka, EventStoreDB, таблица в PostgreSQL)
3. **Перестать хранить «текущее состояние»**: только события
4. **Строить проекции**: читать поток событий и агрегировать в текущее состояние
5. **Использовать снапшоты**: для ускорения восстановления при большом числе событий
6. **Комбинировать с CQRS**: разные проекции для разных read-моделей

## Примеры

- Банковский счёт: события [Открыт(+0), Пополнен(+1000), Снятие(-200)] → текущий баланс = 800
- Корзина: [ItemAdded, ItemRemoved, PromoApplied] → текущая корзина из событий
- Audit для медицинской карты: каждое изменение = событие с временной меткой и автором

## Anti-patterns

- **Event Sourcing везде**: огромный overhead; применять только когда история критична
- **Изменять события**: events иммутабельны — создавать компенсирующее событие, не редактировать
- **Без CQRS**: читать из event store напрямую для каждого запроса — крайне медленно
- **Мелкие технические события**: `FieldUpdated` — не доменные события, не нести бизнес-смысл
