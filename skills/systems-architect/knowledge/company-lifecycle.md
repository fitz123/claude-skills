## Определение

Любой бизнес проходит разные этапы развития, на каждом из которых важны **разные архитектурные характеристики**. Ошибка — проектировать систему для «абстрактного» бизнеса без учёта его текущего этапа. То, что правильно для стартапа, убийственно для растущей компании, и наоборот.

Понимание lifecycle помогает определить **неявные характеристики** (источник 5 из 5-источников метода) и обосновать выбор архитектурного стиля.

## Когда применять

- При поиске неявных архитектурных характеристик
- При выборе архитектурного стиля: стартапу не нужны микросервисы
- При аргументации технических решений бизнесу
- При ревизии архитектуры при переходе компании на новый этап

## Как применять

### Этапы и их ключевые характеристики

**1. Проверка гипотез (Pre-product/Market fit)**

Бизнес ещё не знает, что продаёт. Всё может поменяться завтра: сегодня свитеры из кашемира, завтра — оборудование для стрижки овец. Каждая гипотеза должна проверяться быстро и дёшево.

- **Критичны**: Simplicity, Feasibility, Modifiability (быстро менять под новую гипотезу)
- **Не важны**: Scalability, Availability (ещё нет нагрузки)
- **Архитектура**: монолит, MVP, no-code решения
- **Buy vs Build**: максимум готовых инструментов

**2. Расширение и развитие (Growth)**

Гипотеза подтверждена, есть первые клиенты. Улучшаем существующую систему, наращиваем пользовательскую базу.

- **Критичны**: Scalability, Modifiability, Maintainability, Testability, Agility, Evolvability
- **Появляется**: Deployability (нужно быстро выкатывать фичи)
- **Архитектура**: modular monolith или service-based
- **Happy Cat Box** находился именно здесь → Ибрагим выбрал микросервисы

**3. Захват рынка (Scale)**

Пользователи исчисляются миллионами, выход на новые рынки, многократный рост.

- **Критичны**: Availability, Scalability, Performance, Fault tolerance
- **Важны**: Maintainability (система стала большой и сложной)
- **Архитектура**: микросервисы, распределённые системы, EDA

**4. Зрелость (Mature)**

Стабильный бизнес, размеренный рост, фокус на оптимизацию затрат.

- **Критичны**: Maintainability, Extensibility, Cost efficiency
- **Появляется**: Legacy management (что делать с устаревшими системами)
- **Архитектура**: рефакторинг, strangler fig для замены частей, оптимизация

**5. Закрытие проекта**

- Минимальная стоимость поддержки
- Feasibility для graceful shutdown

### Алгоритм определения lifecycle

1. Выяснить: есть ли подтверждённая гипотеза? Платящие клиенты?
2. Какой темп роста? Насколько стабильны процессы?
3. Какая численность команды и бюджет?
4. Сопоставить с таблицей этапов

## Примеры

**Happy Cat Box (кейс из курса):**
- Гипотеза проверена в Excel-таблицах, есть первая клиентская база
- → **Расширение и развитие**
- → Важны: Scalability, Modifiability, Maintainability, Testability, Agility, Evolvability
- → Плюс явные: Scalability (10К пик) + Time to market (2 спринта)

**Netflix:**
- Начинал с монолита (гипотезы) → перешёл к микросервисам (захват рынка)
- Переход был оправдан: нужны противоречащие характеристики в разных частях системы

**Применительно к КИБЕР-ГЕНПОДРЯД:**
- Стадия: ранний рост (есть первый клиент — Саня, нет ещё крупной базы)
- → Важны: Simplicity (MVP быстро запустить), Modifiability (под разных заказчиков), Maintainability
- → Не нужны сейчас: высокая Scalability (100+ объектов одновременно — не сейчас)
- → Рекомендованная архитектура: modular monolith → при росте добавляем service-based

## Anti-patterns

- **Проектировать для «на вырост»**: закладывать микросервисы для стартапа с 3 разработчиками — убьёт скорость
- **Не пересматривать архитектуру при смене этапа**: то что работало для гипотез, не работает при захвате рынка
- **Игнорировать lifecycle при выборе стиля**: «выберем микросервисы, они лучше» без учёта стадии компании
- **Переоценить текущий этап**: считать себя «захватом рынка» при 100 пользователях → избыточная сложность
- **Недооценить скорость роста**: не заложить возможности для эволюции → придётся переписывать с нуля
