## Определение

**Стратегия** — выбор способа достижения цели: какие элементы включить в систему, как они связаны, какой архитектурный стиль использовать. **Тактика** — конкретные решения в рамках стратегии: какой фреймворк, какая БД, какой язык программирования. Архитектор отвечает за стратегию и задаёт ограничения; тактику делегирует команде.

## Когда применять

- При распределении ответственности в команде: что решает архитектор, что — разработчики
- При написании ADR: описывать «что и почему», а не «как реализовать»
- При code review: архитектор проверяет соответствие стратегии, не синтаксис
- При оценке объёма работы: стратегические решения требуют больше контекста

## Как применять

1. **Определить стратегические решения**: архитектурный стиль, границы контекстов, виды коммуникаций, типы БД
2. **Зафиксировать их в ADR**: с контекстом и обоснованием
3. **Задать ограничения** для команды: «используем async messaging для inter-service communication»
4. **Делегировать тактику**: команда выбирает конкретный брокер, библиотеки, паттерны внутри сервиса
5. **Проверять соответствие** через fitness functions, а не микроменеджмент кода

## Примеры

- **Стратегия**: «используем Event-Driven Architecture для расцепления сервисов» → **Тактика**: «RabbitMQ с exchange fanout»
- **Стратегия**: «каждый контекст имеет свою БД» → **Тактика**: «PostgreSQL для Checkout, MongoDB для Catalog»
- **Плохо**: архитектор диктует названия переменных и структуру классов (тактика, не его ответственность)

## Anti-patterns

- **Тактика без стратегии**: выбрать Kafka, потому что «модно», без анализа характеристик
- **Стратегия без ограничений**: сказать «делайте микросервисы» без правил коммуникации и границ
- **Архитектор-микроменеджер**: контролирует тактические решения команды, убивает самостоятельность
- **Команда без стратегии**: каждый разработчик принимает стратегические решения самостоятельно → хаос
