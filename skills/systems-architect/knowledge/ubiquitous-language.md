## Определение

Ubiquitous Language — единый язык для общения бизнеса и разработки в рамках одного боундед-контекста. Включает не только термины, но и описание процессов, правил, событий. Язык должен использоваться везде: в коде, тестах, документации, разговорах.

## Когда применять

- При старте работы с новым контекстом
- При добавлении нового члена команды
- При разработке требований и тестов
- При написании кода (названия классов, методов, переменных)

## Как применять

1. **Провести сессию с domain expert'ом**: выписать все термины, которые он использует
2. **Создать глоссарий**: каждый термин + определение, синонимы (и запретить использовать синонимы)
3. **Контекстуализировать**: слово «аккаунт» в контексте Billing ≠ «аккаунт» в контексте Auth
4. **Зафиксировать процессы**: описать через Event Storming и sequence diagrams
5. **Написать тесты на Gherkin**: `Given/When/Then` на языке бизнеса, не технических терминов
6. **Использовать в коде**: `class OrderPlaced`, не `class OrderCreatedEvent`

## Примеры

- «Заказ» в контексте склада = набор товаров для сборки; «Заказ» в контексте доставки = маршрут с адресом
- Тест: `When user completes checkout → Then payment is processed` (бизнес-язык, не `POST /api/v1/orders`)
- Глоссарий: «Клиент» = физическое или юридическое лицо, заключившее договор (не «пользователь» и не «покупатель»)

## Anti-patterns

- **Разные термины для одного**: разработчики говорят «user», бизнес — «клиент», тесты — «покупатель» → путаница
- **Технические термины в бизнес-общении**: «мы сделаем таблицу orders» вместо «создадим заказ»
- **Один язык на всю систему**: «клиент» значит разное в разных контекстах → нужен свой язык в каждом
- **Нет глоссария**: язык существует только в головах → при смене людей теряется
