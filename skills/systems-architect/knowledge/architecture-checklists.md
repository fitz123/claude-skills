## Определение

Чек-листы проектирования — структурированная последовательность шагов для работы над архитектурой системы. Помогают не пропустить критические этапы и не переделывать систему по три раза из-за ошибочного порядка действий.

Три главных этапа (идут строго последовательно):
1. **Сбор требований** (розовый)
2. **Анализ** (синий)
3. **Принятие решений** (зелёный)

После каждого этапа — **обязательная валидация со стейкхолдерами**.

## Когда применять

- При старте нового проекта или нового крупного функционала
- При рефакторинге существующей системы
- При онбординге нового архитектора/техлида в проект
- Когда нужно структурировать хаотичный процесс проектирования

## Как применять

### Этап 1 — Сбор требований

- [ ] Определить стейкхолдеров (матрица стейкхолдеров)
- [ ] Провести интервью / воркшоп с каждым стейкхолдером
- [ ] Зафиксировать функциональные требования
- [ ] Выявить явные нефункциональные требования (числа, SLA, ограничения)
- [ ] Выявить неявные нефункциональные требования (из домена, из фраз бизнеса)
- [ ] Определить ограничения (бюджет, команда, сроки, легаси)
- [ ] **✅ Валидация со стейкхолдерами**: все требования поняты правильно?

### Этап 2 — Анализ

- [ ] Event Storming: проработать бизнес-процессы через события/команды/акторов
- [ ] Выделить поддомены (Core / Supporting / Generic)
- [ ] Построить Core Domain Chart: приоритизировать поддомены
- [ ] Определить боундед-контексты и их границы
- [ ] Составить Context Map: как контексты взаимодействуют
- [ ] Создать модель данных для каждого контекста
- [ ] Расставить характеристики качества (quality attributes) для каждого контекста
- [ ] Определить тип коммуникации между контекстами (sync/async)
- [ ] **✅ Валидация со стейкхолдерами**: анализ отражает реальный бизнес?

### Этап 3 — Принятие решений

- [ ] Выбрать архитектурный стиль на основе характеристик и ограничений
- [ ] Выбрать вид базы данных для каждого контекста (SQL/NoSQL/etc)
- [ ] Определить способы коммуникации между сервисами
- [ ] Описать структурную модель системы (сервисы и связи)
- [ ] Зафиксировать ключевые решения как ADR (Architecture Decision Records)
- [ ] Документировать по фреймворку (C4 L1-L2, arc42 или другой)
- [ ] **✅ Валидация со стейкхолдерами**: решения понятны и приняты командой?

### Чек-лист Event Storming сессии

- [ ] Подготовить стикеры: оранжевые (события), синие (команды), жёлтые (акторы), розовые (системы), зелёные (read models)
- [ ] Определить временные рамки и сценарии
- [ ] Начать с бизнес-событий — что происходит в системе?
- [ ] Добавить команды — что вызывает событие?
- [ ] Добавить акторов — кто даёт команду?
- [ ] Выявить агрегаты — какие данные меняются?
- [ ] Найти узкие места и «горячие точки» (hot spots)
- [ ] Выделить boundaries — где меняется словарь?
- [ ] Зафиксировать результат как основу для боундед-контекстов

### Чек-лист модели данных

- [ ] Определить сущности для каждого боундед-контекста
- [ ] Нарисовать атрибуты и типы данных
- [ ] Обозначить связи между сущностями
- [ ] Проверить, что данные не дублируются между контекстами
- [ ] Выбрать вид хранилища (реляционная БД, документная, граф и т.д.) на основе характеристик
- [ ] Определить стратегию миграций
- [ ] Продумать, нужен ли event sourcing

### Чек-лист архитектурной документации

- [ ] C4 L1 — системный контекст (акторы + внешние системы)
- [ ] C4 L2 — контейнеры (сервисы, БД, брокеры)
- [ ] Sequence diagrams для ключевых сценариев
- [ ] ADR для каждого нетривиального решения
- [ ] Глоссарий ubiquitous language
- [ ] Описание характеристик с числами и сценариями
- [ ] README с навигацией по документации

## Примеры

### КИБЕР-ГЕНПОДРЯД — применение чек-листа

**Этап 1 (требования):**
- Стейкхолдеры: рабочий, ИТР, прораб, РПП, РПГ, бухгалтерия
- Явные: «процентовка должна формироваться за 1 день, а не за неделю»
- Неявные: офлайн-работа на объекте (availability ≠ uptime, нужен offline-first)

**Этап 2 (анализ):**
- ES: события — «ГПР утверждён», «работа принята», «КС-2 подписан»
- Core: приёмка работ, процентовка; Supporting: уведомления; Generic: авторизация
- Данные: объект → участок → работа → КС-2/КС-3 (иерархия в рамках контекста «Производство»)

**Этап 3 (решения):**
- Стиль: модульный монолит → эволюция к service-based
- БД: PostgreSQL (реляционная иерархия объект-участок-работа)
- ADR: «Почему не микросервисы сразу» — команда 3 человека, MVP

## Anti-patterns

- **Пропустить валидацию**: перейти к следующему этапу без подтверждения стейкхолдерами → переделки
- **Начать с решений**: сразу рисовать микросервисы без сбора требований и анализа
- **Делать всё одновременно**: ES + модель данных + выбор БД в одну сессию → каша
- **Анализировать в одиночестве**: «ибрагимовский антипаттерн» — уходить в работу без коммуникации
- **Не фиксировать решения**: принять решение устно, без ADR → через 3 месяца никто не помнит почему
- **Насиловать себя при застревании**: если застряли на стейкхолдерах — переключитесь на требования, потом вернитесь
