## Определение

Любое архитектурное решение проходит три этапа: **сбор требований → анализ → принятие решений**. Каждый этап заканчивается обязательной валидацией со стейкхолдерами. Этот процесс не линейный — часто возвращаются на предыдущий этап при новых открытиях.

## Когда применять

- При старте нового проекта или системы
- При значительном рефакторинге/редизайне
- При добавлении новой крупной фичи с архитектурными импликациями
- При смене архитектурного стиля

## Как применять

**Этап 1: Сбор требований**
1. Определить стейкхолдеров и приоритизировать (матрица «Влияние — Интерес»)
2. Собрать консёрны каждой группы стейкхолдеров
3. Собрать функциональные требования (user stories, поведение)
4. Найти внешние ограничения (законы, бюджет, уровень команды)
5. **Валидация** со стейкхолдерами

**Этап 2: Анализ**
1. Определить бизнес-домен и поддомены (Strategic DDD)
2. Классифицировать поддомены: Core / Generic / Supporting
3. Определить боундед-контексты
4. Провести Event Storming — найти поведение
5. Создать модель данных — найти структуру
6. Сравнить контексты из ES с контекстами из DDD
7. Определить архитектурные характеристики
8. **Валидация** со стейкхолдерами

**Этап 3: Принятие решений**
1. Выбрать архитектурный стиль
2. Выбрать виды коммуникаций
3. Выбрать типы БД
4. Определить паттерны для характеристик
5. Описать решения через ADR
6. Определить fitness functions
7. **Валидация** со стейкхолдерами

## Примеры

- Проектирование маркетплейса: Этап 1 = интервью с CEO и командой; Этап 2 = DDD + ES + данные; Этап 3 = service-based + async messaging + ADR
- При открытии нового поддомена на Этапе 2 → возврат к Этапу 1 за дополнительными требованиями

## Anti-patterns

- **Пропустить сбор требований**: «и так понятно что нужно» → строим не то
- **Пропустить валидацию**: решение принято, но стейкхолдеры не согласны → переделывать
- **Один человек все три этапа**: без фидбэка команды теряется перспектива
- **Линейное мышление**: думать что этапы строго последовательны без итераций
