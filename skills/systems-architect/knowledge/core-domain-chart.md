## Определение

Core Domain Chart — визуальный инструмент для классификации поддоменов. Строится на двух осях: **Сложность модели** (Y) и **Влияние на бизнес** (X). Каждый поддомен отображается точкой на диаграмме. Помогает выявить приоритеты и аргументировать их бизнесу.

## Когда применять

- После выявления всех поддоменов — как следующий шаг
- При разговоре с бизнесом о приоритизации инвестиций
- При планировании команд и ресурсов
- При принятии решения buy vs build по каждому поддомену

## Как применять

1. **Нарисовать оси**: Y = «Сложность модели» (низкая → высокая), X = «Влияние на бизнес» (низкое → высокое)
2. **Нанести поддомены** как точки на диаграмму
3. **Интерпретация квадрантов**:
   - Высокое влияние + Высокая сложность → **Core** (максимальный приоритет)
   - Высокое влияние + Низкая сложность → **Supporting** (важно, но не уникально)
   - Низкое влияние + Любая сложность → **Generic** (купить или аутсорсить)
4. **Обсудить с бизнесом**: показать диаграмму, проверить расположение, скорректировать
5. **Принять решения**: Core → инхаус лучшие; Generic → готовое решение

## Примеры

- Uber: алгоритм динамического ценообразования → Core (высокое влияние + высокая сложность)
- Uber: аутентификация пользователей → Generic (купить Auth0 или использовать AWS Cognito)
- Uber: форма обратной связи → Supporting (низкая сложность, умеренное влияние)

## Anti-patterns

- **Строить без бизнеса**: архитектор один расставляет точки без обсуждения с product/CEO
- **Все в Core**: если всё критично → ничто не критично, нет фокуса
- **Не пересматривать**: рынок меняется, что было уникальным — становится commodity
- **Путать сложность реализации с уникальностью**: что-то может быть технически сложным, но не давать конкурентного преимущества
