## Определение

Набор практических советов и паттернов для ведения архитектурной документации и организации работы архитектора. Ответ на вопрос «как работать публично и не потерять контекст».

## Когда применять

- Постоянно, в любом проекте
- Особенно важно при работе с командой (не в одиночку)
- При долгосрочных проектах (>3 месяцев)

## Как применять

### 1. Работайте публично

- Собирайте рабочие документы **в одном месте**, доступном любому сотруднику
- Прикладывайте архитектурное описание и документацию к проекту
- Планируйте **регулярные встречи** с командой: руководством и разработчиками

**Ибрагимовский антипаттерн**: уходить в работу, ни с кем не коммуницировать, ставить команду перед фактом → переделки и конфликты.

### 2. Не забивайте на описание

Описание — это бекап. Оно помогает:
- сэкономить время на объяснении решений
- сохранить контекст для команды и себя в будущем

**Практические советы:**
- Делайте **частые и короткие пометки с контекстом** — везде: в моделях, документах, заметках
- Ведите **дневник проекта**: текстовый файл с датой и тем, что происходит прямо сейчас. Помогает вспомнить, что «PM Маша три месяца назад хотела фичу X»
- Пишите **не «что», а «почему»** — описывайте контекст, а не дублируйте код/схему словами

### 3. «Почему» важнее диаграмм

Итоговое решение поменяется много раз. А ответ на «почему» помогает людям, которые придут разбираться позже.

**Классический антипаттерн в коде:**
```python
# increment counter by 1
counter += 1
```
Хороший комментарий: «Увеличиваем счётчик после каждой успешной попытки авторизации — защита от брутфорса».

### 4. Храните версии моделей и описаний

- **Текст**: используйте git или онлайн-редактор с аудит-логом изменений
- **Модели**: вкладки в диаграммном инструменте, каждая = версия за дату
- В Miro/Figjam нет версионирования → используйте **фреймы** для разных версий
- В Lucid — версии есть встроенные

### 5. Переключайтесь при застревании

Если зависли → не насилуйте себя:
- Тупите с Event Storming → займитесь моделью данных, потом вернётесь
- Застряли на стейкхолдерах → переключитесь на требования или ограничения

### 6. Не меняйте то, что работает

Самое главное правило для рабочих проектов: **не трогайте работающее без веской причины**.

### 7. Проверяйте экономику изменений

Если видите «неправильную» архитектуру — **сначала поговорите с руководством**. Может оказаться, что исправление дороже предполагаемой пользы.

### 8. Учитывайте жизненный цикл проекта

- **Новый стартап**: максимальная изоляция для каждой гипотезы/фичи, не тратьте время на анализ всего проекта — бизнес сам ещё не знает что нужно
- **Зрелый продукт**: можно и нужно полный анализ

### 9. Не предлагайте решение в первые минуты

При добавлении нового функционала — соблазн предложить техническое решение сразу. Скорее всего, оно ошибочное. Алгоритм:
1. Остановиться
2. Разобраться в проекте поглубже
3. Ответить на «что нужно сделать»
4. Только потом думать «как делать»

### 10. Оставляйте артефакты работы

Чем больше оставите и подробнее опишете → тем проще коллегам разобраться + тем проще вам найти, что пошло не так.

**Типичные артефакты:**
- Event Storming (фото/экспорт стикеров)
- Модели данных
- Context Map
- ADR (Architecture Decision Records)
- Глоссарий (Ubiquitous Language)
- Структурные диаграммы (C4 L1-L2)

## Примеры

### КИБЕР-ГЕНПОДРЯД — практики документирования

**Дневник проекта** (пример записи):
```
2026-02-15: Обсудили с Саней структуру ГПР. Решили, что ГПР — это иерархия:
Объект → Этап → Вид работ → Операция. Саня сказал, что в реальной жизни
этапы могут перекрываться по времени — нужно учесть в модели данных.
```

**«Почему» в ADR:**
> ADR-003: Почему модульный монолит, а не микросервисы — команда 2 человека,
> деплой в одно место проще, нет сетевых задержек внутри. Пересмотрим при
> росте команды до 5+ человек.

**Версионирование моделей:**
- Figma: фреймы «v1 - MVP», «v2 - Phase 2», «current»
- Git: документация рядом с кодом в `/docs/architecture/`

## Anti-patterns

- **Молчаливый архитектор**: анализировать в одиночестве, показывать только результат → нет доверия, нет feedback loop
- **Документировать «что»**: «сервис принимает запрос и возвращает ответ» — бесполезно; нужно «почему такой интерфейс»
- **Один большой документ**: монолитная документация устаревает и никто не читает; лучше маленькие атомарные ADR
- **Не версионировать**: «где та диаграмма с прошлого спринта?» → нет истории → нельзя откатиться
- **Документировать после**: «опишем потом» → никогда; делайте пометки во время работы
