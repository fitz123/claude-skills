## Определение

Декомпозиция системы — процесс определения, какие боундед-контексты выделять в отдельные сервисы, а какие оставлять вместе. Главное правило: **если характеристики и ограничения двух элементов можно объединить — объединяйте. Если противоречат друг другу или невыполнимы вместе — разделяйте.** Декомпозиция — не разбивка "по модулям" или "по функциям", а следствие разных характеристик и ограничений.

## Когда применять

- При выборе между service-based и микросервисной архитектурой
- При разделении монолита на сервисы
- При добавлении нового функционала с новыми характеристиками
- Когда монолит "болит" из-за конфликтующих требований к масштабированию

## Как применять

**Подготовка:**
1. Выписать все боундед-контексты (минимальные атомарные элементы)
2. Выписать характеристики для каждого контекста
3. Выписать внешние ограничения для каждого контекста
4. Начать с "одного монолита" — все BC вместе

**Итеративный процесс разделения:**

Для каждой итерации задавать вопрос: "Есть ли BC с уникальным ограничением или характеристикой, несовместимой с остальными?"

**Критерии выделения в отдельный сервис:**
- **Уникальное внешнее ограничение** (финансовый комплаенс → иммутабельная БД)
- **Внешняя система** (стороннее решение — не разрабатываем сами)
- **Противоречащий релизный цикл** (один контекст деплоится ежедневно, остальные — раз в 2 недели)
- **Противоречащие нагрузочные характеристики** (одна часть требует elasticity, другие — нет)
- **Уникальный монолитный стиль** (часть требует Pipeline или Microkernel — остальным не подходит)
- **Изоляция риска** (высокая надёжность нужна только в одном месте → не распространять на всю систему)

**Критерии объединения в один сервис:**
- Одинаковые характеристики и ограничения
- Одинаковый релизный цикл
- Тесная доменная связь (нет смысла в network hop)
- Совместное масштабирование (одна нагрузка)

**Фиксация стоимости**: учитывайте, что два сервиса = дороже одного. Иногда "несовместимые" характеристики дешевле держать в одном сервисе с компромиссом.

**После разделения — выбор монолитного стиля для каждого сервиса:**

```
Для каждого сервиса выбрать внутренний монолитный стиль:
- Нужна расширяемость через плагины? → Microkernel
- Последовательная обработка данных? → Pipeline
- Нужна хорошая модульность + сложная логика? → Modular Monolith
- Всё остальное → Layered
```

## Примеры

**Happy Cat Box — 6 итераций:**

| Итерация | Что выделили | Причина |
|---|---|---|
| 1 | Биллинг | Финансовый комплаенс → иммутабельность данных, нельзя применить ко всей системе |
| 2 | Алиэкспресс + Доставка | Внешние системы, разрабатывать не нужно |
| 3 | Определение параметров игрушек | Уникальные характеристики: simplicity + reliability, изолировать чтобы не тормозить общий релиз |
| 4 | Магазин + Мэтчинг (вместе) | Elasticity + scalability уникальны; характеристики у обоих одинаковые → в один сервис |
| 5 | Тестирование игрушек | Релизный цикл 1 день vs. 2 недели у остальных |
| 6 | Склад / Инфлюенсеры (раздельно) | Разные поддомены, маркетинг часто меняет логику → перестраховка |

**КИБЕР-ГЕНПОДРЯД — пример декомпозиции:**

| Итерация | Что выделить | Причина |
|---|---|---|
| 1 | Процентовка (КС-2, КС-3) | Иммутабельность финансовых данных + юридические требования к форматам |
| 2 | Уведомления | Generic-поддомен, может использовать готовые решения (Firebase, Expo) |
| 3 | ГПР (планирование) | Только у РПП/РПГ, высокая сложность, частые изменения логики планирования |
| 4 | Учёт выполненных работ | Offline-first требование уникально для мобильного клиента рабочих |
| - | Справочники + Роли (вместе) | Одинаковые характеристики: consistency, редко меняются |

## Anti-patterns

- **Выделять по функциям/слоям**: "сервис для уведомлений" только потому что "это отдельная функция" — нужны уникальные характеристики
- **Дробить без необходимости**: два сервиса с одинаковыми характеристиками = лишняя сложность
- **"Микросервисный монолит"**: сервисы есть, но общая БД и тесный coupling → худший из миров
- **Игнорировать стоимость разделения**: инфраструктура двух сервисов + их коммуникация = значительный overhead
- **Разделять по команде "потому что две команды"**: Conway's Law — учитывать, но не делать главным критерием
- **Не выбирать монолитный стиль для каждого сервиса**: разработчики без ориентиров → Big Ball of Mud
