## Определение

При распиле монолита или проектировании микросервисов нужно выбрать, **где провести границу** между сервисами. Чаще всего используют бизнес-концепции (bounded context, поддомен), но иногда этого недостаточно.

Если характеристики внутри одного bounded context расходятся (например, одна команда требует 1k rps, другая — 1.5 пользователя в час), нужен другой способ разделения.

**Воронка разделения:** от общего к частному
```
Домен → Поддомен → Bounded Context → Бизнес-команда → Чтение/Запись
```

## 4 стратегии выбора границ

### 1. По бизнес-концепциям (основная)

**Когда:** в большинстве случаев (9 из 10 по опыту Антона). Разделение по bounded context или поддоменам.

**Как проверить:** провести быструю ES-сессию. Если элементы содержат много требований, а команд и событий меньше половины — возможно, нужна другая стратегия.

**Hint:** если сомневаешься, сделай несколько вариантов и выбери с **наименьшим количеством связей** между элементами.

---

### 2. По техническим шагам

**Когда:** система состоит из реализации базового неизменяемого технического функционала, который переиспользуется везде.

**Примеры:**
- Сервис ресайза картинок
- Работа с POP3/SMTP/IMAP протоколами
- Сервис PDF-генерации

**Плюс:** явных связей будет больше, но скрытых — меньше, что снижает глобальную сложность.

**Нотификации — тонкая грань:**
- Создание сообщения (бизнес-логика) → нельзя выносить в технический сервис
- Протокол отправки (SMTP, FCM) → можно вынести как технический шаг

---

### 3. По данным (Data-first)

**Когда:** система состоит из работы с данными без сложной логики и процессов.

**Примеры:**
- CRUD-сервисы для справочников
- Сервисы репликации данных

**Осторожно:** это путь к entity-service антипаттерну. Используй только когда данные действительно главная ценность, без бизнес-логики.

---

### 4. По пользовательским ролям

**Когда:** разные роли пользователей работают с принципиально разными процессами и данными.

**Пример:** отдельный сервис для администраторов, менеджеров, тестировщиков.

**Когда НЕ нужно:** если роли работают с одними и теми же данными и процессами — разделять по ролям неправильно.

---

### Дополнительно: по CQRS (чтение/запись)

**Когда:** внутри одного bounded context чтение и запись имеют принципиально разные характеристики.

**Пример из курса:** для записи — максимальная consistency (реляционная БД), для чтения — availability (key-value или колоночная). Нужно разделить базу на запись и чтение.

```
Write Service → Реляционная БД (consistency)
             ↓ репликация/события
Read Service  → Key-Value БД (availability, performance)
```

## Алгоритм выбора стратегии

```
1. Начни с бизнес-концепций (bounded context/поддомен)
   → Проведи ES-сессию
   → Если элементы = много требований + мало команд → рассмотри другую стратегию

2. Проверь конфликты характеристик
   → Одна команда: 1k rps, другая: 2 запроса в день → нужно разделить

3. Если система = только данные без логики → по данным
4. Если система = технические функции (resize, SMTP) → по техническим шагам
5. Если роли работают с разными процессами → по ролям
6. Если чтение и запись требуют разных гарантий → CQRS
```

## Как проверить правильность границ

- **Меньше связей = лучше**. Сделай несколько вариантов разделения, сравни количество связей
- **Конфликтующие характеристики в разных сервисах** — это хороший признак правильных границ
- **Instability** — посчитай для каждого варианта, важные элементы должны стремиться к I=0
- **ES-сессия**: если элементы «встали» в понятные группы — границы правильные

## Примеры

### Happy Cat Box — тестирование игрушек

**Проблема:** генерация отчётов — 1k rps, остальное тестирование — 1.5 кота в час.

**Решение:**
- Нельзя держать в одном сервисе — конфликт характеристик
- Разделить на 3: генерация отчётов, тестирование, расчёт зарплат
- Генерация отчётов читает из read-only реплики БД тестирования (допустимый лаг 1 минута)

### КИБЕР-ГЕНПОДРЯД

**Разделение модуля приёмки:**

| Стратегия | Результат |
|---|---|
| По бизнес-командам | Фиксация факта → верификация объёма → подписание КС-2 |
| По ролям | Портал рабочего, портал ИТР, портал прораба (если интерфейсы разные) |
| CQRS | Запись приёмки (consistency) + чтение для отчётов РПП (availability) |

**Рекомендуемое:** по бизнес-командам (основной путь) + CQRS если нужны отчёты в реальном времени.

## Anti-patterns

- **«Всегда по бизнес-процессам»** — 90% случаев да, но при конфликте характеристик нужно смотреть глубже
- **«Всегда по техническим шагам»** — ведёт к entity-service или notification antipattern
- **«Разделить всё по ролям»** — роли часто работают с теми же данными → лишние связи
- **Не считать связи** — «кажется правильным» не считается. Сделай несколько вариантов и посчитай
- **Атомарный уровень = всегда бизнес-команда** — иногда нужно спуститься до чтение/запись (CQRS)
