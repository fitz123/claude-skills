## Определение

Saga — паттерн управления распределёнными транзакциями через цепочку локальных транзакций. Если один шаг не удался, выполняются компенсирующие транзакции для отмены предыдущих шагов. Альтернатива двухфазным коммитам (2PC) в микросервисах.

## Когда применять

- Нужна «транзакция» через несколько микросервисов
- Каждый сервис имеет свою БД (нет глобального ACID)
- Eventual consistency допустима
- Нужна возможность компенсации (отмена уже выполненных шагов)

## Как применять

**Два стиля:**

**Choreography** (через события):
1. Каждый сервис слушает событие → выполняет шаг → публикует следующее событие
2. Нет центрального координатора
3. Плюсы: простота, decoupling; Минусы: сложно отследить состояние саги

**Orchestration** (через оркестратор):
1. Центральный сервис-оркестратор координирует шаги
2. Оркестратор вызывает каждый сервис и ждёт результата
3. Плюсы: явный workflow, легко отладить; Минусы: coupling к оркестратору

**Шаги:**
1. Определить шаги транзакции
2. Для каждого шага определить компенсирующую операцию
3. Выбрать стиль: choreography или orchestration
4. Реализовать идемпотентность каждого шага

## Примеры

- Оформление заказа: [Создать заказ] → [Списать деньги] → [Списать со склада] → [Создать доставку]
- Компенсация при отказе на шаге 3: [Отменить заказ] ← [Вернуть деньги] ← [Откат]
- Orchestration: Order Service оркестрирует Payment, Inventory, Shipping

## Anti-patterns

- **Saga вместо простого запроса**: если транзакция укладывается в один сервис → не нужна Saga
- **Нет компенсирующих транзакций**: «откатить» просто нельзя → потеря данных
- **Нет идемпотентности**: дубль вызова → двойное списание
- **2PC в микросервисах**: распределённый lock → производительность и deadlocks
