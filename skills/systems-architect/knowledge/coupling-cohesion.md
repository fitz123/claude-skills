## Определение

**Coupling** (внешняя связанность) — сила связей между элементами системы. **Cohesion** (внутренняя связанность) — сила связей внутри одного элемента. Золотое правило проектирования: стремиться к **низкому coupling** и **высокому cohesion**, чтобы изменения в одном элементе не ломали другие и каждый элемент решал одну связную задачу.

## Когда применять

- При проектировании границ сервисов и модулей
- При анализе Event Storming модели: найти концентрации связей
- При ревью архитектуры: посчитать coupling между контекстами
- При рефакторинге: понять, что стоит объединить, что разделить
- При выборе между монолитом и микросервисами

## Как применять

1. **Проанализировать coupling**: подсчитать все виды связей между контекстами (10 видов)
2. **Если coupling высокий между A и B**: рассмотреть объединение в один контекст
3. **Проанализировать cohesion**: что находится внутри элемента, связано ли это одной целью?
4. **Если cohesion низкий**: разделить элемент по ответственностям
5. **Проверить через instability metric**: I = Ce / (Ca + Ce)

## Примеры

- **Высокий cohesion**: сервис «Оплата» содержит только логику проведения платежей
- **Низкий cohesion**: сервис «Оплата» содержит логику платежей + уведомления + генерацию отчётов → нужно делить
- **Низкий coupling**: сервисы общаются через события, не напрямую
- **Высокий coupling**: сервис A вызывает 8 методов сервиса B синхронно + читает напрямую его БД

## Anti-patterns

- **Coupling = 0**: полная изолированность = не система. Какой-то coupling необходим
- **Принудительная декомпозиция**: делить всё на мелкие части ради «микросервисности», не думая о cohesion
- **God service**: один сервис с низким cohesion — делает всё подряд
- **Игнорировать виды coupling**: видеть только HTTP-вызовы, не замечать shared database или temporal coupling
