## Определение

**Локальная сложность** — сложность внутри одного компонента (размер кодовой базы, количество бизнес-правил). **Глобальная сложность** — сложность взаимодействия между компонентами (координация, консистентность, observability). Микросервисы уменьшают локальную сложность, но увеличивают глобальную. Монолит — наоборот. Выбор — это осознанный trade-off.

## Когда применять

- При выборе архитектурного стиля
- При оценке стоимости декомпозиции монолита
- При обосновании решения перед руководством
- При сравнении альтернатив: что сложнее — поддерживать монолит или оркестрировать 50 сервисов?

## Как применять

1. **Оценить текущую локальную сложность**: насколько большой и запутанный монолит/сервис?
2. **Оценить текущую глобальную сложность**: сколько сервисов, каково их взаимодействие?
3. **Определить узкое место**: что болит больше — локальная или глобальная сложность?
4. **Рассчитать trade-off**: декомпозиция снизит локальную, но вырастет глобальная — выгодно ли?
5. **Учитывать зрелость команды**: глобальная сложность требует DevOps, observability, распределённых транзакций

## Примеры

- Стартап с 3 разработчиками: монолит → локальная сложность терпима, глобальная = 0 ✅
- 50 микросервисов у команды из 5 человек: глобальная сложность убивает продуктивность ❌
- Modular Monolith: компромисс — логическое разделение без глобальной сложности распределённых систем

## Anti-patterns

- **Микросервисы как дефолт**: декомпозировать без оценки, вырастает глобальная сложность без выгоды
- **Монолит как догма**: не декомпозировать когда локальная сложность уже непосильна команде
- **Игнорировать глобальную сложность**: «просто добавим ещё один сервис» — в итоге распределённый монолит
- **Не учитывать зрелость команды**: микросервисы без DevOps культуры = катастрофа
