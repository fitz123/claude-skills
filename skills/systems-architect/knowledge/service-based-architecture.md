## Определение

Service-Based Architecture — распределённый стиль с крупными сервисами (по 2-5 на систему) и общей БД. Промежуточный вариант между монолитом и микросервисами. Считается **самым прагматичным** распределённым стилем: даёт основные преимущества распределённых систем без их полной сложности.

## Когда применять

- Нужны распределённые характеристики (независимый деплой, отдельные команды)
- Команда ещё не готова к полным микросервисам
- Нет потребности в очень гранулярном масштабировании
- Хотите разделить монолит без полной декомпозиции

## Как применять

1. **Разделить по крупным доменам**: 2-5 сервисов, каждый = группа связанных боундед-контекстов
2. **Оставить общую БД**: или несколько БД на группу сервисов
3. **Определить API между сервисами**: REST/gRPC для синхронной коммуникации
4. **Добавить API Gateway** для внешнего трафика
5. **Масштабировать по нагрузке**: каждый сервис деплоится независимо

**Характеристики:**
- Simplicity/Cost: ★★★
- Deployability: ★★★★
- Scalability: ★★★
- Fault Tolerance: ★★★★

## Примеры

- E-commerce: сервис «Каталог и поиск», сервис «Заказы и оплата», сервис «Доставка»
- SaaS-платформа: сервис «Аутентификация», сервис «Контент», сервис «Биллинг»

## Anti-patterns

- **Общая БД с прямым coupling**: сервис А читает таблицы сервиса Б напрямую → скрытый coupling
- **Слишком мелкие сервисы**: превращается в микросервисы с общей БД — худший из миров
- **Нет API между сервисами**: прямые вызовы к БД → через resources coupling
- **Использовать когда нужен один сервис**: если монолит справляется — не усложняйте
