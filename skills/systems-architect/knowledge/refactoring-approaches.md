## Определение

Пять основных задач, с которыми сталкиваешься при модернизации распределённых систем. Каждая требует своего подхода к анализу и реализации. Ибрагим решал их последовательно в Happy Cat Box.

## Задача 1: Добавление новой функциональности

**Проблема:** бизнес хочет новый функционал. Куда его добавить?

**Алгоритм анализа:**
1. Сделай Event Storming для нового функционала
2. Сравни ES-модели: похожа ли новая функция на уже существующую?
3. Определи характеристики нового функционала (из поддомена + специфические требования)
4. Сравни характеристики с существующими bounded contexts

**Варианты:**
- **Расширить похожий существующий сервис** — если боундед контексты совпадают
- **Создать идентичный новый сервис** — если боундед контексты разные, даже при похожей реализации сейчас

**Золотое правило:** не объединяй только потому, что сейчас похоже. Два bounded context из разных поддоменов разойдутся — и поддерживать их в одном сервисе станет невозможно.

**Как выбирать между вариантами:**
- Сравни по плюсам и минусам
- Если равнозначно — сравни по рискам (что случится, если риск реализуется?)
- Спроси бизнес — финансисты/менеджеры знают, будет ли у поддоменов общее будущее

## Задача 2: Восстановление характеристик существующего сервиса

**Проблема:** сервис перестал удовлетворять требованиям из-за:
- Изменения нагрузки (DoS от соседних сервисов)
- Смены поддомена (generic → core)
- Изменения требований стейкхолдеров

**Алгоритм:**
1. Определи, какая характеристика нарушена (performance, modifiability, etc.)
2. Выбери из трёх вариантов:
   - **Поменять архитектурный стиль** — если стиль не соответствует новым характеристикам
   - **Улучшить характеристику в текущем стиле** — рефакторинг, но без смены архитектуры
   - **Изменить способ коммуникации** — с синхронного на асинхронный (часто самый эффективный)

**Выбор способа внедрения изменений:**

| | Менять «на горячую» | Создать новую версию рядом |
|---|---|---|
| **Когда подходит** | Не меняется архитектурный стиль и схема данных | Меняется стиль, данные или нужен другой язык |
| **Плюс** | Не нужно замораживать фичи | Всегда есть rollback |
| **Минус** | Сложный rollback | Нужно перегонять данные |

**Золотое правило Антона:** «Стратегия с дублированием функций может оказаться дешевле». Добавить новый «цех» рядом и отключить старый — часто выгоднее, чем рефакторить работающий.

## Задача 3: Объединение технических сервисов

**Проблема:** технические сервисы (нотификации, gateway) создают нестабильность важных сервисов.

**Алгоритм:**
1. Посчитай instability всех сервисов: I = Ce/(Ca+Ce)
2. Проверь: важные core-сервисы должны стремиться к I=0
3. Если технический сервис делает важные сервисы нестабильными — убирай его
4. Перенеси логику в каждый bounded context

**Пример — сервис нотификаций:**
- С отдельным сервисом: core-сервисы I=1 (нестабильны), нотификации I=0 (стабильны)
- Без отдельного сервиса: все сервисы I=0 (стабильны)

**Вывод:** Нотификации — технический шаг реализации бизнес-команды, а не отдельный сервис. Как «отобразить результат оплаты на странице» не делают отдельным сервисом, так и нотификации не нужно изолировать.

## Задача 4: Вынос сервиса из «монолита» (распил)

**Паттерны декомпозиции — выбор по 4 вопросам:**

1. Нужна ли новая база данных?
2. Нужно ли менять модель данных?
3. Надо ли использовать другой язык?
4. Нужно ли менять реализацию бизнес-процессов?

```
Нет новой БД, нет смены модели:
  → Strangler Fig или Tactical Forking (зависит от языка)

Нужна новая БД или смена модели данных:
  → Change Data Capture (Data First)
```

**Чек-лист — стоит ли вообще распиливать:**
- ✅ Есть смысл, если при распиле улучшатся характеристики
- ❌ Нет смысла, если у всех команд в bounded context одинаковые характеристики
- ❌ Нет смысла, если у всех bounded context одинаковые характеристики в домене
- ❌ Нет смысла, если у всех поддоменов одинаковые характеристики (монолит дешевле!)

## Задача 5: Исправление Entity-сервиса

**Проблема:** entity-сервис — единственный источник правды для данных, высокая связанность, высокая instability.

**Алгоритм:**
1. Посчитай instability entity-сервиса
2. Найди «владельца» данных (кто мутирует сущность?)
3. Объедини entity-сервис с его владельцем
4. Переведи чтение данных на event-driven (стриминг событий об изменении)
5. Каждый потребитель хранит свою локальную копию нужных данных

**Результат:** entity-сервис исчезает. Данные живут в сервисе-владельце и стримятся в event bus.

## Применение в КИБЕР-ГЕНПОДРЯД

| Задача | Пример |
|--------|--------|
| Добавление функционала | Добавить модуль процентовки: отдельный bounded context в поддомене финансов |
| Восстановление характеристик | Если ГПР перегружает систему — перевести на асинхронный event-driven |
| Объединение технических сервисов | Не делать отдельный «сервис уведомлений» — встроить в каждый модуль |
| Вынос из монолита | Распилить монолит приёмки: сначала generic-поддомены, потом core |
| Исправление entity-сервиса | Не делать «сервис объектов строительства» — данные владеет тот, кто их создаёт |

## Anti-patterns

- **Распил ради распила** — без улучшения характеристик это просто трата ресурсов
- **Начинать с нотификаций** — популярный «первый кандидат» на вынос, но создаёт каплинг
- **Объединять похожие bounded context** — если поддомены разные, логика разойдётся
- **Рефакторить «на горячую» при смене стиля** — без rollback-стратегии это опасно
