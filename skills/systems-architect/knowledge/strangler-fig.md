## Определение

Strangler Fig — паттерн постепенной миграции: новая система строится рядом со старой и постепенно «обвивает» её, перехватывая функциональность. Аналогия с деревом-душителем (strangler fig), которое растёт вокруг хозяина. Альтернатива Big Bang-переписыванию.

## Когда применять

- При модернизации legacy-монолита
- При переходе от монолита к микросервисам
- Когда нельзя остановить production и переписать всё сразу
- Когда старая система должна продолжать работать параллельно

## Как применять

1. **Выбрать первый кандидат**: контекст с наибольшей ценностью или наименьшим риском
2. **Построить фасад/прокси**: новые запросы идут через прокси → старая система
3. **Создать новый сервис** для выбранного контекста параллельно
4. **Переключить трафик**: прокси перенаправляет запросы контекста в новый сервис
5. **Удалить код** из старой системы после стабилизации
6. **Повторить** для следующего контекста

## Примеры

- Монолит интернет-магазина: сначала вынести Catalog (читающий сервис, безопаснее), потом Checkout, потом Orders
- Каждый новый сервис проходит: параллельная работа → shadow mode → переключение → cleanup
- API Gateway как точка переключения: `/api/catalog/*` → новый сервис, остальное → монолит

## 3 канонических шага (из курса)

Strangler Fig Application как Code First паттерн работает в три шага:
1. **Создать новый сервис** рядом со старым с gateway-прослойкой
2. **Итеративно переносить** команды из ES из старого сервиса в новый
3. **Завершить**, когда все команды перенесены → удалить старый сервис

Gateway-прослойка скрывает от внешнего мира детали переноса: снаружи виден один интерфейс, внутри — два сервиса.

## Сравнение с другими паттернами декомпозиции

| Критерий | Strangler Fig | Tactical Forking | Change Data Capture |
|---|---|---|---|
| Подход | Code First | Code First | Data First |
| Язык | Любой (можно менять) | Тот же стек | Любой |
| Нужна новая БД | ❌ | ❌ | ✅ |
| Принцип | Перенос команд итеративно | Копировать + удалять лишнее | Сначала данные, потом логика |
| Rollback | Легкий (proxy переключает) | Есть (старый код жив) | Есть (параллельная работа) |

**Выбор SFA vs Tactical Forking:** если нужен другой язык программирования — Strangler Fig. Если тот же стек и удалять легче, чем выносить — Tactical Forking.

## Anti-patterns

- **Big Bang**: переписать всё сразу → риск потери данных, долгий стоп для бизнеса
- **Не удалять старый код**: «задушенная» функциональность остаётся в монолите → дублирование
- **Начинать с самого сложного**: Core-контекст как первый кандидат → высокий риск
- **Без фасада**: менять монолит изнутри без прокси → хаотичная миграция
- **Менять данные одновременно с кодом**: двойная сложность. Если нужна смена БД — используй CDC вместо SFA
