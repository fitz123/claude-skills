Microkernel
Иногда называется plug-in architecture. Придумали много лет назад. Состоит из двух 
частей: core system и плагинов (в английской литературе называются plug-in 
components). 
Core system содержит минимальную функциональность, которая необходима для 
запуска приложения. Плагины — это изолированные и независимые компоненты, 
которые включают в себя процессинг, фичи и кастомный код, который нужен для 
улучшения core-части.
Бизнес-логика делится между core system и плагинами, что улучшает extensibility, 
adaptability и isolation характеристик для новых фич.
Коммуникации между core system и плагинами реализуются через два способа: 
• Point-to-point. Это вызов кода в одном инстансе. Реализуется либо через прямой 
вызов функции, либо через очередь, реализованную в коде, например Queue из 
стандартной библиотеки. При таком виде коммуникаций компоненты 
реализуются как shared libraries, package (из java и js) или namespaces (c#/f#);
• REST и async messaging. В этом случае каждый плагин выступает отдельным 
сервисом, вызывающимся асинхронно через событие или синхронно через http/

Pipeline
Монолитный стиль. Может называться pipes and ﬁlters. Принципы его архитектуры 
можно найти в Unix-way (bash/sh/zsh), а также в функциональном 
программировании. На нём построен MapReduce.
Состоит из ﬁlters и pipes.
Filters (фильтры) — это изолированные stateless-компоненты, каждый из которых 
выполняет одну задачу. Можно делать композицию из нескольких фильтров.
Есть четыре вида фильтров:
• producer (иногда называют source) — это начальная точка процесса;
• transformer получает данные, трансформирует, потом отправляет дальше. В 
функциональном программировании это функция map;
• tester принимает данные, которые по критерию или нескольким отправляются 
дальше. В ФП — функция reduce;
• consumer завершает пайплайн — сохраняет конечный результат в БД или 
отображает результат в интерфейсе.
Pipes (трубы, пайпы) — это коммуникационный канал между фильтрами, который 
принимает данные с выхода одного фильтра и отправляет на вход следующего. 
Через трубы можно передавать любые данные. Чаще всего передача данных 
однонаправленная.

Популярный асинхронный распределённый архитектурный стиль, который 
необходим для приложений с highly scalable и high-performance. Адаптивный: 
подходит для мелких, больших и сложных приложений, целой системы и её частей.
Основная идея заключается в реагировании на ситуации и выполнении действий, 
основанных на событиях, с помощью event processor (consumers), которые 
асинхронно продьюсят и консьюмят события. При этом все коммуникации являются 
асинхронными событийными, даже между клиентом и приложением. Дальше будет 
рассказано, почему это только асинхронные коммуникации.
Есть две реализации: хореографическая (broker topology) и оркестрируемая 
(mediator topology). Подробнее можно почитать в книге Fundamentals of Software 
Architecture.

У нас есть ограничение: определение параметров игрушек должно быть пошаговым,

Чем больше в системе элементов, тем сложнее добиться её надёжности (reliability), 
потому что в любой сложной системе выше риск, что что-то пойдёт не так: отвалится, 
будет некорректно работать или забудется. Молоток надёжнее автомобиля, потому 
что проще.

весь монолит поддерживать эту характеристику, то, скорее всего, каждый релиз 
будет тестироваться месяцами, что сломает двухнедельный релиз. Поэтому, чтобы 
система релизилась быстро, нужно изолировать боундед-контекст определения 
параметров игрушек и делать высокий reliability конкретно для него.

📌Нюансы
Считается, что синхронные коммуникации подходят в трёх случаях:
. Нужно как можно быстрее отправить текущее состояние данных в другое 
место.
. Когда состояние данных должно быть актуально в другом месте.
. Есть сервисы, которым нужно быстро получать ответы от второго сервиса.
А асинхронные коммуникации нужны, когда:
. Данные не обязательно получать моментально.
. Существует большой поток данных для обработки, а обработка должна 
контролироваться по нагрузке.
. Нужно одни и те же данные отправить в несколько мест одновременно.
В реальной жизни в 80% случаев подходят оба вида коммуникаций. Но если 
система использует синхронные коммуникации и выбор вида однозначный 
— используйте синхронные, чтобы не тратить время на подготовку 
инфраструктуры для асинхронных.
Кроме этого, у синхронных блокирующих вызовов есть нюанс, связанный с 
«передачей» некоторых характеристик от клиента к серверу. Допустим, в 
системе есть ЛМС и сервис с информацией об учениках и ЛМС должен 
возвращать информацию обо всех курсах всех учеников за секунду. Если 
для генерации этой информации нужно сделать синхронный блокирующий 
вызов в сервис учеников, то на него распространится ограничение по 
скорости выполнения запроса. Получается, за одну секунду уже должен 
отрабатывать не один сервис, а два. Даже если нет проблем с реализацией 
такого требования, остаётся неявная связанность — её легко упустить, и

Architectural decision records, ADR
Это документ, который описывает специфичное решение, связанное с архитектурой 
и дизайном системы. Он хорош тем, что сохраняет контекст принятия решения, даже 
если в нём были скрытые условия, не было подходящих технологий или архитектор 
не знал о существовании иного решения проблемы. Без этой информации систему 
сложно развивать, потому что каждый новый сотрудник компании будет тратить 
время на поиск ответов или переделывать систему без учёта скрытых условий.
ADR состоит из короткого текстового файла на 1-2 страницы с обязательными 
полями. Инструмент может быть любым: вики, notion, conﬂuence, гугл-док. 
Документирование контекста важнее инструмента. 
Мы советуем заполнять такие поля (примеры в отдельном доке ниже, пока просто 
пробегитесь по полям):
Title. Короткое описание архитектурного решения и номер документа. Нумерация 
обязательна для сохранения порядка принятых решений.
Status. Нужен для отслеживания решений. Есть четыре базовых статуса:
• Proposed. Ждёт одобрения начальником или архитектурным комитетом;
• Accepted. Принято и идёт на реализацию;
• Rejected. Отклонено. Важно описать причину;
• Superseded. Говорит о том, что ADR был заменён другим ADR. Нужно, если старое 
решение уже неактуально. Например, вначале выбрали монолит, а потом 
оказалось, что характеристики поменялись, и теперь нужны микросервисы. В 
самом статусе важно указать номер нового ADR, чтобы сохранить исторический 
контекст и получить линки.
Context. Главная секция документа, в которой описываются причины и условия 
принятия решения. Здесь же указываются альтернативные решения, а их анализ — в 
дополнительной секции Alternatives.
Decision. Описание решения и его обоснование. Здесь важно делать акцент на 
«почему», а не «как» и описывать причины вместо описания того, как решение 
работает.
Consequences. Описание влияния принятого решения на систему. Если умеете в 
анализ trade-oﬀ, то эта секция подойдёт для описания.
